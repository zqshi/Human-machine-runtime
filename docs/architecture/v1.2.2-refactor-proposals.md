## P1-5：presentation → infrastructure 直连重构（最大架构债）

### 现状（系统性违规，20+ 处）

前端 `presentation` 层大量直接 import `infrastructure` API 客户端，绕过 `application` 用例层。违反 CLAUDE.md §1.2「presentation 直接调用 infrastructure = 跳过 application 层 = 违规」。命中（节选）：

| 文件 | 行 | 直调客户端 |
|------|----|-----------|
| `presentation/features/studio/MarketplacePage.tsx` | 14-15 | `marketplaceApi` / `upstreamMarketplaceApi` |
| `presentation/features/studio/create/McpDatabaseFlow.tsx` | 14-15 | `toolApi` / `credentialManagementApi` |
| `presentation/features/studio/create/AgentCreateFlow.tsx` | 23,30,38 | `agentDefinitionApi` |
| `presentation/features/admin/Credential*.tsx`（4 文件） | — | `credentialManagementApi` |
| `presentation/features/admin/ToolApprovalsSection.tsx` | 7 | `v19AdminApi` |
| `presentation/features/admin/FeatureFlagSection.tsx` | 9 | `v19AdminApi` |
| `presentation/features/admin/RuntimeTemplatesSection.tsx` | 12 | `v19AdminApi` |
| `presentation/features/admin/AgentAnalyticsPage.tsx` | 11 | `analyticsApi` |

### 根因

项目未引入 application 用例层抽象（CLAUDE.md §2.3 备注：effect + fetch 数据加载模式，未引入 React Query/Suspense）。每接一个真 API（去 mock 工作），就多一处直调 infrastructure，application 层形同虚设。

### 方案

引入 application 用例 hook（use case hook）作为 presentation ↔ infrastructure 之间的薄层：

```
presentation (组件)
   ↓ 调用
application/use-cases/*.ts  (useCredentialUseCase / useAgentDefinitionUseCase ...)
   ↓ 调用
infrastructure/api/*.ts  (现有 API 客户端不变)
```

每个 use case hook 封装：数据加载（effect）+ 状态管理（zustand 或 local state）+ 错误处理 + 业务规则（如 McpDatabaseFlow 的探测→勾选→发布流程编排）。

**分批策略**（避免一次性大改）：
1. 批次 1（P1 优先）：`McpDatabaseFlow` / `AgentCreateFlow` / `Credential*`（核心创建流程）
2. 批次 2：`MarketplacePage` / `AgentAnalyticsPage`
3. 批次 3：admin 子组件（ToolApprovals/FeatureFlag/RuntimeTemplates）

### 影响范围

- 新增 `application/use-cases/` 目录（~10 个 hook 文件）
- 改 20+ presentation 文件的 import（从直调 infrastructure 改为调 use case hook）
- 现有 infrastructure API 客户端不变（接口稳定）
- 测试：use case hook 加组件测试（非快照）

### 排期

3-5 天，独立版本/任务，**不混入功能开发**（§4.3）。建议作为 v1.2.3 或独立重构版本。

## P1-6b：tool-management-service 拆分（639 行）

### 现状

`server/src/contexts/tool-management/tool-management-service.ts` 639 行，单方法均 < 80 行红线（最大 `syncDatabase` 77 行），属「文件膨胀」非「方法膨胀」。命中 §12 信号2（P1，§12.2 下一版本必修）。

方法清单（23 个 public + 4 个 private）：

| 职责 | 方法 | 行段 |
|------|------|------|
| Source 管理 | listSources/getSource/createSource/updateSource/deleteSource | 71-148 |
| Sync | syncSource + syncOpenApi/syncDatabase/syncGateway（private） | 150-422 |
| Introspect | introspectSource | 424-456 |
| 连接测试 | testConnection | 458-498 |
| Definition 管理 | listDefinitions/getDefinition/updateDefinition/toggleDefinition | 502-522 |
| Instance 绑定 | listInstances/bindTool/unbindTool | 524-548 |
| 执行 | executeTool | 550-601 |
| 统计 | getStats/getCallLogs | 605-625 |
| 其他 | uploadSpec/resolveCredential(private) | 60-70, 629-639 |

构造依赖：`sourceRepo` / `definitionRepo` / `instanceRepo` / `callLogRepo` / `credentialSecretProvider` + 内部 new 的 `OpenApiParser`/`DbIntrospector`/`GatewayDiscoverer`。

### 拆分难点

- `resolveCredential`（private，用 credentialSecretProvider）被 Source 创建 / Sync / Introspect / Execution 共享 → 拆子 service 时需重复持有或提取共享 helper
- `syncXxx` 共享 `sourceRepo` + `definitionRepo` + parser/discoverer
- `executeTool` 共享 `definitionRepo` + `sourceRepo` + `callLogRepo` + `resolveCredential`

### 方案（委托模式，接口不变）

`ToolManagementService` 保留所有 public 方法签名（调用方 routes + 17 测试不变），内部委托 4 个子 service：

```
ToolManagementService（薄委托层，~120 行）
  ├─ ToolSourceService     （Source 管理 + Sync + Introspect + testConnection，~280 行）
  ├─ ToolDefinitionService （Definition 管理，~60 行）
  ├─ ToolBindingService    （Instance 绑定，~40 行）
  └─ ToolExecutionService  （executeTool + resolveCredential 共享，~80 行）
```

- `resolveCredential` 提取为共享 helper（或 ToolExecutionService 持有，其他 service 注入）
- 子 service 各自持有所需 repo，构造时由 ToolManagementService 组装传入

### 影响范围

- 新建 4 个子 service 文件（`tool-source-service.ts` / `tool-definition-service.ts` / `tool-binding-service.ts` / `tool-execution-service.ts`）
- 改 `ToolManagementService` 为委托层（public 方法保留，内部 `this.sourceService.xxx()`）
- `bootstrap.ts` 组装处不变（仍 new ToolManagementService，内部自组装子 service）
- 测试：17 用例不变（接口不变），可补子 service 单测

### 排期

1-1.5 天。属 §12.2「下一版本必修」，建议 v1.2.3。低风险（委托模式 + 测试保护），但仍应独立任务不混入功能开发。

## P1-7：跨聚合访问解耦

### 现状

application service 直接 import 其他 context 的 service，绕过聚合边界。命中 §12 信号5（P1）：

| 文件 | 行 | 跨聚合引用 |
|------|----|-----------|
| `contexts/agent-core/application/studio-service.ts` | 17-18 | `shared-assets/skill-service` + `shared-assets/domain/shared-skill` |
| `contexts/agent-core/application/agent-definition-service.ts` | 2 | `audit-observability/audit-service` |
| `contexts/tool-management/application/approval-gate.ts` | 4 | `system-config/system-config-service` |

### 方案

引入领域事件或端口接口解耦，不直接引其他 context 的 service：

**选项 A（领域事件，推荐）**：被依赖方发布事件，依赖方订阅。
- `studio-service` 不直接调 `skill-service`，而是订阅 `skill.requested` 事件
- 适合异步、最终一致场景

**选项 B（端口接口）**：在 agent-core domain 定义 `ISkillPort` / `IAuditPort` / `ISystemConfigPort` 接口，bootstrap 时由被依赖方 context 适配注入。
- 复用 `assembly-provider.ts` 的 port 模式（IInstanceLookupPort / IAgentDefinitionPort 等）
- 适合同步调用场景

**混合**：同步用端口接口（如 audit 写日志），异步用事件（如 skill 召回）。

### 影响范围

- agent-core domain 新增 port 接口（ISkillPort / IAuditPort / ISystemConfigPort）
- bootstrap.ts 装配端口适配（类似 `adaptBoundTools` / `adaptInstanceLookup`）
- 3 个 application service 改为依赖 port 接口
- 被依赖方 context（shared-assets / audit-observability / system-config）不改

### 排期

2-3 天，§12.2 P1 下一版本必修。与 P1-5 正交可并行。
