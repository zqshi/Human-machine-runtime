# HMR 多租户架构方案

> 产品视角驱动的多租户设计，定义边界优先于技术实现。

---

## 一、核心产品定义

### 1.1 租户定义

| 问题 | 定义 |
|------|------|
| 一个租户 = ？ | 一个企业客户（如"某银行"、"某电商"） |
| 租户内用户 | 该企业的 IT 管理员、业务运营、审计员、终端员工 |
| 租户间关系 | 完全独立，互不可见（标准 SaaS 隔离） |
| 平台运营商 | HMR 团队自身，拥有跨租户全局管理视图 |

### 1.2 租户拥有的资源

```
租户（企业客户）
├── 数字员工实例（数量受 Plan 配额限制）
├── 技能库（私有 + 从市场安装的）
├── 知识库（私有文档/语料）
├── AI 网关配置（模型白名单、Key、用量）
├── 用户账号（管理员/运营/审计/终端用户）
├── 审计日志（只能看自己的）
└── 配额与计费（Token 预算、实例数、存储）
```

### 1.3 共享 vs 隔离策略

| 资源 | 隔离策略 | 说明 |
|------|---------|------|
| 数字员工实例 | **严格隔离** | A 租户不可见 B 的员工 |
| 技能/工具 | **分层** | 平台公共市场 + 租户私有开发 |
| 知识库 | **严格隔离** | 企业数据绝对不能跨租户 |
| AI 模型/网关 | **配置隔离** | 共享基础设施，租户级配额/白名单 |
| 审计日志 | **严格隔离** | 按 tenantId 物理分区 |
| 系统配置 | **平台全局** | 租户不可见 |

## 二、产品分层模型

```
┌─────────────────────────────────────────────────────┐
│  L0  平台运营商（HMR 团队）                           │
│  看到：所有租户、全局配额、计费、基础设施监控           │
├─────────────────────────────────────────────────────┤
│  L1  租户管理员（客户 IT 负责人）                      │
│  看到：本租户一切 — 员工、技能、知识库、用量、用户       │
│  能做：CRUD 数字员工、配置 AI 网关、管理本租户用户      │
├─────────────────────────────────────────────────────┤
│  L2  租户运营（客户业务人员）                          │
│  看到：被授权的数字员工、任务执行情况                   │
│  能做：与数字员工交互、下发任务、查看结果               │
├─────────────────────────────────────────────────────┤
│  L3  终端用户（客户普通员工）                          │
│  看到：分配给自己的数字员工                            │
│  能做：对话、审批、协作                               │
└─────────────────────────────────────────────────────┘
```

## 四、租户生命周期

```
创建 → Provisioning → 活跃 → [冻结] → [归档/删除]
```

| 阶段 | 触发 | 产品行为 |
|------|------|---------|
| **创建** | 运营商手动 / 自助注册 / 上游同步 | 分配 slug、默认配额、初始管理员 |
| **Provisioning** | 创建后自动 | 初始化知识库空间、分配 AI 网关 namespace |
| **活跃** | 正常使用 | 全功能可用，配额实时校验 |
| **冻结** | 欠费 / 违规 / 手动 | 只读，不能创建新实例，现有实例暂停 |
| **归档** | 手动 / 合同到期 | 数据保留 N 天后物理删除，不可恢复 |

### 产品决策点

- 冻结时现有数字员工：给 72h 宽限期，期间只读，超时自动暂停
- 归档后数据保留：30 天内可恢复，之后物理删除
- 租户开通方式：MVP 阶段运营商手动创建，后续支持自助注册

## 五、租户入口设计

### 方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **子域名** `bank-a.hmr.ai` | 品牌感强、天然隔离 | 证书管理复杂、DNS 配置 | 大客户 / ToB SaaS |
| **URL 前缀** `/t/bank-a/...` | 实现简单、单域名 | 没有品牌感 | 内部平台 / MVP |
| **登录时绑定** | 最简单 | 多租户用户切换麻烦 | 用户只属于一个租户 |
| **Token 隐式绑定** | 用户无感 | 跨租户切换不友好 | 当前方案 |

### 推荐策略

- **短期**：登录时强制绑定 tenantId（不允许 fallback）
- **中期**：SSO 自动映射组织 → 租户
- **长期**：子域名 + 自定义域名支持

## 六、配额与计费模型

### 6.1 配额维度

```
租户配额
├── 实例数上限（数字员工数量）
├── 并发实例上限
├── 用户数上限
├── Token 预算（月/日）
├── API 调用频率（次/分钟）
├── 存储容量（知识库 + 文档，MB）
├── 模型访问白名单
└── 功能开关（知识库/AI网关/自定义工具）
```

### 6.2 Plan 分级

| 计划 | 最大实例 | 最大用户 | 月 Token 预算 | API 日调用 | 存储 |
|------|---------|---------|-------------|----------|------|
| free | 3 | 5 | 100K | 1,000 | 1GB |
| standard | 10 | 50 | 1M | 10,000 | 10GB |
| enterprise | 100 | 500 | 10M | 100,000 | 100GB |

### 6.3 计费触发点

| 事件 | 计量方式 | 超额策略 |
|------|---------|---------|
| AI 模型调用 | Token 数（input + output） | 软限：降级到低成本模型 |
| 知识库存储 | MB/月 | 硬限：禁止新增 |
| 实例运行时间 | Pod 小时 | 硬限：禁止创建新实例 |
| API 调用 | 次数/天 | 软限：限速降级 |

> **实现状态(v1.0 骨架)**：`contexts/billing/` 已建立记账骨架：
> - 表：`billing_events`(事件流水) / `billing_accounts`(租户余额) / `billing_invoices`(预留)
> - 服务：`BillingService.recordEvent` 仅记账,**不生成账单**(账单生成留作后续迭代)
> - 埋点：`ClaudeAgentSdkAdapter.onTaskComplete` 成功分支记 `token_usage` 事件(按定价表估算 USD 成本),决策审批状态变更记 `decision_closed`(未触发)
> - 暴露：`GET /platform/billing/events` / `GET /platform/billing/account` / `POST /platform/billing/events`

### 6.4 产品决策点

- 超额处理：日级软限（告警+降级），月级硬限（禁止操作）
- 定价模式：固定套餐 + 超额按量阶梯
- 临时加量：支持管理员一键购买加量包

## 七、技能市场模型

### 7.1 分层结构

```
平台公共市场（所有租户可见）
├── 官方技能（平台运营商维护）
├── 社区技能（第三方贡献）
└── 安装 → 复制到租户私有空间

租户私有空间（只有本租户可见）
├── 自研技能
├── 从市场安装后的定制版本
└── 租户专属 prompt/workflow
```

### 7.2 产品决策点

| 问题 | 决策 |
|------|------|
| 租户能否向市场发布技能？ | Phase 2 支持，需审核流程 |
| 安装后是否跟随上游更新？ | Fork 快照，手动选择是否升级 |
| 运行数据是否回流作者？ | 仅聚合统计（安装数），不含业务数据 |

## 八、安全与合规

### 8.1 安全边界

| 要求 | 实现方式 |
|------|---------|
| 数据不出租户边界 | Scoped Repository / PostgreSQL RLS |
| 审计可追溯 | 所有操作记录 who/when/what + tenantId |
| 租户数据可导出 | GDPR 合规，提供 data export API |
| 租户数据可删除 | 归档后完全清除，不留残余 |
| 管理员操作需二次确认 | 删除租户、批量操作需审批流 |
| 跨租户操作审计加强 | 平台运营商查看租户数据需留痕 |

### 8.2 隔离保障机制

| 层级 | 机制 |
|------|------|
| 数据库层 | PostgreSQL RLS 或 Scoped Repository |
| 应用层 | tenantContext 中间件强制注入，拒绝空 tenantId |
| API 层 | 所有查询接口必须带 tenantId 参数 |
| 测试层 | 跨租户不可见性自动化测试 |

## 九、技术实现要点

### 9.1 租户上下文传播链路

```
请求入口
  → authMiddleware（解析 JWT/Session）
  → tenantContextMiddleware（强制提取 tenantId，空值 → 403）
  → c.set('tenantId', tenantId)
  → Handler（从上下文取 tenantId）
  → ScopedRepository（构造时绑定 tenantId，所有查询自动过滤）
```

### 9.2 数据隔离方案选择

| 方案 | 隔离强度 | 复杂度 | 适用规模 |
|------|---------|--------|---------|
| **行级隔离 + Scoped Repo** | 中 | 低 | < 100 租户 |
| **PostgreSQL RLS** | 高 | 中 | 100~1000 租户 |
| **Schema 隔离** | 高 | 高 | 大客户定制场景 |
| **物理库隔离** | 最高 | 最高 | 金融/合规强制要求 |

**推荐**：行级隔离 + Scoped Repository（当前阶段）→ 后续按需升级 RLS。

### 9.3 路由层对应关系

| API 路径 | 面向角色 | 租户感知 |
|---------|---------|---------|
| `/api/platform/*` | L0 平台运营商 | 跨租户全局视图，tenantId 作为查询参数 |
| `/api/admin/*` | L1 租户管理员 | 从 JWT 自动绑定 tenantId，强制隔离 |
| `/api/control/*` | L1~L2 | 从 JWT 自动绑定 tenantId |
| `/api/cockpit/*` | L2~L3 终端用户 | 从 JWT 自动绑定 tenantId |

## 十、落地路线图

### Phase 1：单租户加固（当前 → 2 周）

- [ ] 消灭所有 `'default'` fallback，改为登录时强制绑定
- [ ] 增加 `tenantContextMiddleware`，空 tenantId 返回 403
- [ ] Repository 层增加跨租户不可见性测试
- [ ] 审计日志查询强制传 tenantId

### Phase 2：多租户 MVP（有第二个客户时，4~6 周）

- [ ] 运管平台租户 CRUD 实际使用（创建、配额配置、冻结/激活）
- [ ] 登录流程支持租户选择或 SSO 自动映射
- [ ] AI Gateway 按租户分 namespace（LiteLLM team/key）
- [ ] 知识库按租户物理隔离 collection
- [ ] 技能市场区分"公共"和"租户私有"

### Phase 3：商业化（规模化，8~12 周）

- [ ] 子域名 / 自定义域名支持
- [ ] 计费系统对接（用量采集 + 账单生成）
- [ ] 租户自助注册 + 试用期
- [ ] 数据导出 / 删除 API（合规）
- [ ] SLA 分级（不同 Plan 不同可用性保障）
- [ ] 升级到 PostgreSQL RLS

## 十一、当前代码差距总结

| 维度 | 标准多租户 | 当前状态 | 差距 |
|------|-----------|---------|------|
| 数据隔离强制性 | RLS / Scoped Repository | 可选参数，靠人记 | 高 |
| 租户上下文传播 | 中间件自动注入全链路 | Handler 手动取 | 中 |
| 空租户处理 | 拒绝请求 / 报错 | fallback 到 default | 高 |
| 服务配置隔离 | 按租户分配额/key/模型 | 全局共享 | 高 |
| 前端入口隔离 | subdomain / prefix | 无 | 低（短期不需要） |
| 多租户测试覆盖 | 跨租户不可见性测试 | 只有单租户 seed | 中 |
