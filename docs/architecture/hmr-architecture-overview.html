HMR Architecture Overview

面向多渠道决策指挥的中间层运行时

HMR（Human-machine-runtime，人机运行时）不是 IM、不是 Agent 框架、不是项目管理工具，而是坐在它们之上的 中间层操作系统：聚合多渠道消息 → AI 智能分拣 → 人机协同决策 → 命令下达到执行闭环追踪。 后端为单体 Hono + DDD 限界上下文（29 个 context 同进程，60+ 装配组件），前端为单 SPA 三入口， 通过 AgentRuntimeAdapter 与 ChannelAdapter 两道标准接口实现上层 Agent 框架与下层渠道的双向可插拔。

代码库：monorepo（server + client-suite） 后端：Hono + TS + Drizzle（单进程 29 context） 前端：React + zustand + Tailwind（Apple HIG） 当前版本：v1.8 已归档 / v1.2.2 待启动 更新时间：2026-06-24

1. 一句话总结

HMR 把消息归一化、优先级评估、跨渠道去重、决策建议、执行回执这五件事做成了不可替代的中间层。 上层 Agent 框架（OpenClaw / Claude Agent SDK / Dify / Coze）可换、下层渠道（Matrix / WPS / 钉钉 / 飞书 / 企微）可换， 运行时层不动。这是跨平台政治护城河——钉钉不会帮聚合飞书，只有中立第三方能做。

三层解耦 核心

Agent 框架可插拔（AgentRuntimeAdapter）、渠道可插拔（ChannelAdapter），运行时层消息归一化 + 决策引擎 + 执行编排不可替代。

单体 DDD 29 context

不是 29 个微服务。单 Hono 进程内按限界上下文组织，domain 零外部依赖，依赖单向向右。避免微服务运维成本，保留领域隔离。

claude-worker 沙箱 执行

Agent 任务在独立 Docker 容器内跑 Claude Agent SDK，按行 NDJSON 上报进度，预算熔断 + 会话续接 + Token 用量入账。

per-instance 模型隔离 授权

ModelGrantChecker 白名单（off/log/enforce 三态灰度）+ LlmKeySyncService 把授权同步为 LiteLLM virtual key，per-instance 兜底隔离。

双模式前端 三入口

单 SPA 三入口：main（用户端）/ admin（租户管理）/ ops（运管）。appMode 切换 IM/OpenClaw 两套 Dock。

声明/运行 reconcile 云原生

v1.8 把实例升级为 desiredState(actual)/specGeneration diff 增量调和，reconciler 主动调和、health-monitor 被动探活，每 5 分钟并行。

3. 后端 DDD 分层与 29 限界上下文

后端按限界上下文组织（非顶层四层目录）。每个 context 内部按需划分 domain（纯逻辑、零外部依赖）/ application（用例编排）/ adapters（外部适配）。 依赖方向单向：domain → application → adapters，禁止反向。路由层只做参数校验与转发，业务判断下沉。

核心域 身份与租户

  • identity-access — JWT + SSO(OIDC) + RBAC + AuthProviderRegistry（local/OIDC/platform-be 多 provider），scope 区分 platform/tenant
  • tenant-management — 租户生命周期 + 套餐（Free/Pro/Team）+ PlanService
  • tenant-instance — 数字员工实例，provisioners/ 三实现（composite / container-orchestrator / local），desiredState/actual 声明运行分离
  • quota-management — 配额四维度（容量/资源/AI/数据）+ 读时检查 + QuotaMonitor 周期巡检

业务域 运行与编排

  • runtime-engine — 消息归一化 / 优先级 / 去重 / 推荐 / 回执五引擎 + DecisionProjector
  • agent-core — Agent 执行，session/harness/sandbox/domain 四子层 + AdapterRegistry
  • channel — ChannelAdapter 接口 + 三适配器（matrix/wps/websocket）+ InboundPipeline + ChannelRouter + DecisionConsole + ChannelBridgeService
  • shared-agent / shared-assets — 跨租户共享 Agent / 技能工具资产
  • knowledge + document — 知识库 + 文档（WeKnora 检索，可选启用）
  • employee-memory — 数字员工记忆（mem0，可选启用）
  • workspace / marketplace / agent-profile — 对接 xspace / clawhub / portal
  • analytics — 用量统计 + AnalyticsService

支撑域 平台能力

  • gateway — 7 个上游 client + 5 条 proxy 路由 + ModelGrantChecker + LlmKeySyncService + GatewayHealth 周期探活
  • credential-vault — 凭证加密存储 + LeaseService 短期租约 + CredentialManagementService
  • tool-management + mcp-management — 工具资产 + MCP executor（McpService）
  • scheduler — 定时任务（8 handler + pg-advisory-lock 防并发 + cron-calculator）
  • notification + push-channel — 邮件通知 + 推送通道
  • audit-observability + observability — 审计日志 + 指标追踪 + TokenUsageService + TraceSyncJob
  • system-config — 系统配置（SSO 开关、模型授权灰度模式等）
  • eval-benchmark — 评测基准（真实 agent trajectory 评测，EvalService + EvalAgentInvoker）
  • billing — 计费骨架（事件累加；写侧 v1.2.2 补全）
  • department — 部门组织

4. 前端架构

client-suite 是 npm workspaces：主应用 apps/web + 设计 token 包 packages/ui-tokens。 apps/web/src 严格四层：domain（15 子域纯逻辑）→ infrastructure（API client + Matrix + channels）→ application（28 个 zustand stores + services + hooks）→ presentation（25 个 feature 模块）。

三入口与双模式

三个 HTML 入口分别渲染 App / AdminApp / OpsApp，启动前都调 bootstrapApp() 注入 session 过期回调（清 authStore）。 App.tsx 检测 URL：?code&state → OIDC 回调（SSOCallback）；?loginToken → Matrix SSO 回调；否则按 isLoggedIn 切 LoginPage/WorkspacePage。 uiStore 持有 appMode（'im'/'openclaw'），Dock 据此渲染两套 item：IM 模式（消息/轻应用/通讯录…+ 工作面板切换键）、OpenClaw 模式（工作面板/Agent/Team…）。 sharedAgentChatService 在 IM 模式内复用 useAgentChat，幂等 initConversation 准备会话状态机。

设计语言

Apple HIG glass morphism，主色 #007AFF。暗色模式经 [data-mode="openclaw"] CSS 变量覆盖。

API 适配层

infrastructure/api/ 24 个 client/adapter，openclawApiAdapter 收敛决策中心调用，sessionHandler 统一处理 401 过期，与后端六域路由对齐。

stores（节选）

authStore / openclawStore / agentStore / chatStore / marketplaceStore / uiStore 等 28 个，一个 store 一个文件。

已知技术债 D8/D9/D10：useAgentChat 仍直接 import openclawStore，完整 AgentRuntimePort 抽象未做， Agent 定义与运行时未彻底分离（影响替换为 Hermes 等框架）。marketplace「对话」半成品（mk-* 与 sa-* id 体系不兼容）。 IM 模式流式回包待浏览器真实验证。

5. 核心运行时引擎

运行时层是 HMR 的价值核心，分三块：runtime-engine（消息→决策链路）、agent-core（Agent 执行）、channel（渠道入站编排）。

runtime-engine · 消息决策五引擎

  • MessageNormalizer — 多渠道消息归一化：意图分类、紧急度评估、实体抽取
  • PriorityScorer — 多因子优先级（意图×紧急度×发送者权重×时效性）
  • DedupEngine — 跨渠道去重（Jaro-Winkler 相似度 + 指纹哈希）
  • RecommendationEngine — 决策建议（历史模式匹配 + 多源上下文聚合）
  • ReceiptManager — 执行回执路由（成功/失败/进度回传原始渠道）
  • DecisionProjector — 决策投射

agent-core · Session/Harness/Sandbox/Domain 四子层

  • Session — session-store.ts 会话状态
  • Harness — harness.ts 编排 + litellm-llm-client + budget-guard 预算熔断
  • Sandbox — adapter-registry + claude-agent-sdk-adapter（主）+ openclaw-adapter（降级）+ infrastructure/
  • domain — agent-executor / agent-definition / assembly-provider（按定义组装工具+skill）/ rag-context-provider / trace-recorder / pricing

5.1 channel · 入站流水线与路由

channel context 不只是 adapter 集合，而是完整的入站编排链：

  • InboundPipeline — 责任链，use() 注册 handler，逐个执行；任一 handler 抛错中止整条流水线防止不一致状态传播
  • ChannelRouter — 按 RoutingRule.priority 降序匹配条件，resolveTargetChannels 决定消息投递到哪些渠道；无匹配 fallback 到 websocket
  • DecisionConsole — 决策控台，组合 ChannelService + ChannelRouter，产出 DispatchCommand（targetChannels + roomId + message）
  • ChannelBridgeService — 桥接 Matrix Bot 与运行时

5.2 claude-worker 沙箱执行协议

infra/claude-worker/worker.ts 是独立 Docker 容器内的 worker：接收 CLAUDE_TASK_JSON 环境变量， 调用 Claude Agent SDK query()，按行 stdout 输出 NDJSON 事件供宿主 DockerWorkerRunner 消费。 

{"type":"session_id","sessionId":"sess_xxx"}      # 会话建立（首次或 resume）
{"type":"progress","progress":50,"usage":{...}}     # 进度 + Token 用量
{"type":"result","result":"...","stopReason":"end_turn"}  # 最终结果
{"type":"done","sessionId":"sess_xxx"}               # 任务完成
{"type":"error","message":"..."}                     # 失败（exit 1）

退出码：0 完成 / 1 故障或超时 / 2 CLAUDE_TASK_JSON 缺失。支持会话续接（sessionId resume）、 预算熔断（budget-guard，CLAUDE_DEFAULT_BUDGET_USD）、RAG 上下文块与 skill 上下文块前置注入。 ANTHROPIC_API_KEY 未配置时不注册 adapter，系统降级到 OpenClaw adapter。

6. 模型授权与 per-instance 隔离

HMR 在 LLM 调用上做了双层隔离，确保数字员工实例只能用被授权的模型：

ModelGrantChecker 白名单

校验「某 instance 是否被授权使用某 model」，授权关系存于 instance_model_grants 表（白名单语义，默认关闭）。

灰度三态（SystemConfigService.aiGateway.enforceModelGrants 控制）：

  • off — 不校验直接放行（默认）
  • log — 校验并记录未授权，仍放行（可观测期）
  • enforce — 校验并拒绝未授权（强制期，返回 403）

LlmKeySyncService 兜底

把 instance 的 grants 同步为 LiteLLM virtual key：grants 变更时，计算被授权的全部模型别名，在 LiteLLM 生成/更新一把绑定 allowed_models 的 key，缓存到 instance_llm_keys。

chat.ts 调用时用该 key，实现 per-instance 模型隔离（LiteLLM 层兜底，即使应用层漏校验也被 LiteLLM 拦）。

同步失败不阻断主流程（grants 仍写成功），仅记录 sync_status=failed；LiteLLM 未配置时跳过生成但记 stale。

chat.ts（/api/openclaw/chat）是用户端对话端点：优先调 LiteLLM，不可用时自动返回 mock 回复（保证前端全链路可演示，不依赖外部服务）。DEFAULT_MODEL = claude-sonnet-4-6。

8. 声明/运行 reconcile 与调度

v1.8 把数字员工实例升级为云原生声明式模型，scheduler context 周期调度两条兜底路径：

instance-reconciler 主动调和

  • 每 5 分钟遍历实例，按 desired→actual + specGeneration diff 调和
  • decideReconcileAction 返回 noop 时跳过（避免无谓 get/save）
  • 有 drift 调 InstanceService.reconcile：状态调和（stop/start）+ spec 增量
  • reconcile 失败累计达阈值（默认 3）→ 内部 rebuild 兜底（整机重建）
  • 职责：声明态(desired/spec) → 运行态(actual)

instance-health-monitor 被动探活

  • 每 5 分钟探活远端真实态（remote）
  • unhealthy 先 force-reconcile 轻量恢复
  • 持续不健康累积 → rebuild 兜底（30min cooldown 去重）
  • 职责：运行态(actual) → 远端真实态(remote)
两者每 5 分钟并行、互不阻塞，共用 InstanceService.rebuild。理论可能短时双 rebuild，由 cooldown 去重 + rebuild 幂等保护。 pg-advisory-lock 保证同一任务在多实例部署时不并发执行。

多租户数据隔离

两域：Platform（platform scope，全局可见，跨租户管理）与 Tenant（tenant scope，tenantId 强制过滤）。 配额四维度：容量（实例/并发/用户）、资源（CPU/内存/存储）、AI（Token/调用/速率）、数据（保留期/Webhook/知识库）。

LiteLLM 隔离

所有大模型调用经 LITELLM_BASE_URL 中转，per-instance virtual key 兜底。换模型仅改 LiteLLM 配置，HMR 链路无感。

私有化部署

内网无 Anthropic 出口时，ANTHROPIC_BASE_URL 经企业代理（LiteLLM /v1/messages 或自建）转发。见 docs/architecture/private-deployment-paths.md。

13. 团队心智模型

不要把 user / tenant / channel / agent / runtime / tool / skill / knowledge / credential 混在一起。

User 是人；Tenant/Workspace 是组织归属；Channel 是外部入口（可换）；Agent 是产品定义（可换框架）； Runtime 是运行载体（desiredState/actual 分离）；Tool 是可调用能力契约；Skill 是可复用技能包； Knowledge 是可检索知识边界；Credential 是安全材料。它们可以关联，但必须通过服务 API 显式绑定。

问题应该问哪个 context
这个登录用户、外部 sender 对应谁？权限域？identity-access（AuthProviderRegistry）
这个租户、workspace、套餐、配额是什么？tenant-management / quota-management
这个数字员工实例由谁承载？desired/actual 状态？tenant-instance（reconciler + health-monitor）
某条消息如何归一化、优先级、去重、决策建议？runtime-engine
入站消息怎么编排、路由到哪个渠道？channel（InboundPipeline + ChannelRouter + DecisionConsole）
Agent 任务怎么执行？预算熔断？Token 入账？agent-core（Harness + budget-guard + claude-worker）
某 instance 被授权用哪些模型？怎么隔离？gateway（ModelGrantChecker + LlmKeySyncService）
换成 Dify/Coze/自研 Agent 框架怎么接？agent-core/sandbox 的 AgentRuntimeAdapter
接入钉钉/飞书/企微/邮件渠道？channel/adapters 的 ChannelAdapter
Agent 绑定了哪些技能、工具？怎么组装？shared-assets / tool-management + assembly-provider
某个 Agent/Skill 绑定了哪些知识？怎么检索？knowledge（WeKnora）+ rag-context-provider
数字员工的长期记忆？employee-memory（mem0）
外部密钥、租约、加密存储？credential-vault
对接 clawhub/portal/xspace/claw-farm/LiteLLM？gateway/clients + gateway/routes proxy
定时任务、reconcile/health 巡检？scheduler（8 handler + pg-advisory-lock）
一次调用链路、Token 用量在哪里查？observability / analytics / trace-recorder
计费、对账、用量报表？billing / analytics（v1.2.2 补全写侧）
