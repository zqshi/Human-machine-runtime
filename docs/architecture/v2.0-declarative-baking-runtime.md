## 0. 背景与现状实证 `[IMPLEMENTED]`

经四维实证（非记忆、非推断），当前 HMR 架构是"**声明式 + 运行时动态组装**"，三维度判定：

| 维度 | 状态 | 证据 |
|---|---|---|
| 声明式 | ✅ 已建立 | `AgentDefinitionSpec`（agent-definition.ts:81）7 维声明 + `validateAgentDefinitionSpec` 校验（:117）+ generation 世代版本号（:100）+ 前端 7 步向导 |
| 编译固化 | ❌ 完全缺失 | `grep compile/bake/固化/freeze/seal` 全空；`generation` 是数据代际号非编译产物；无 build-time 把声明固化为不可变产物的机制 |
| 运行分离 | ⚠️ 半成品 | adapter-registry 按 framework 路由（dispatchTask，adapter-registry.ts:40）；但投产主路径 tool-loop 执行在 server 进程内；sandbox 执行器是模块级单例二选一（executor-factory.ts:23），无法并存 |

**运行时漂移证据**：`harness.ts:179-244` 每次 dispatch 都调 `assemblyProvider.assemble()`（动态批查 boundTools/boundSkills 解析，assembly-provider.ts:122）+ `personaProvider.getPersona()`（动态取 systemPrompt/guardrails，harness.ts:228）→ DB 查询+映射开销 + 运行时漂移风险（线上 Agent 行为不可锁定，每次 dispatch 重新解析）。

**升级目标**：补齐编译固化层，让"声明态产物"（prompt/guardrails/tools/skills/配额/路由）在发布时固化为不可变 RuntimeManifest，运行时只组装"运行时上下文"（凭证/session/实时配额/输入）；CubeSandbox 作为高安全可选执行层与 OpenSandbox 并存。

## 1. 架构总览 `[IMPLEMENTED]`

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 1 声明层(已有,扩展)                                    │
│  AgentDefinitionSpec(7维) + generation + 校验                │
│  sandboxTemplate 加 'kvm-microvm'(路由 CubeSandbox)          │
└──────────────────┬──────────────────────────────────────────┘
                   │ 发布/激活 + generation bump 触发 bake
                   ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 2 编译固化层(新增,核心)                                │
│  BakingService.bake(agentDefinitionId, generation)          │
│  → RuntimeManifest(不可变):compiledSystemPrompt/guardrails/ │
│    tools/skillsContext/quota/runtimeRoute/sandboxStrategy    │
│  复用 assembleTools/assembleSkills(不重发明)                 │
│  落 agent_runtime_manifests 表(唯一约束 agentDefinitionId+  │
│  generation,status=baked 只读)                              │
└──────────────────┬──────────────────────────────────────────┘
                   │ 运行时读 manifest(有则用,无则降级老路径)
                   ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 3 运行时动态组装层(改造,灰度)                          │
│  harness 读 RuntimeManifest 拿声明态产物(快/不可变/可信)     │
│  运行时仍动态组装:凭证解密/session/实时配额/用户输入/trace   │
│  ─────────────────────────────────────────────────────────  │
│  运行分离:                                                   │
│   adapter-registry 按 runtimeRoute 路由(tool-loop/claude)    │
│   SandboxRouter 按 sandboxStrategy 路由执行后端             │
│     ├─ OpenSandboxExecutor(docker,默认/轻量)               │
│     └─ CubeSandboxExecutor(KVM MicroVM,高安全)             │
└─────────────────────────────────────────────────────────────┘
```

**边界原则**：声明态产物 = 固化（manifest，不可变）；运行时上下文 = 动态组装（每次 dispatch 实时）。两者正交，不可混淆。

## 3. Layer 2 编译固化层（新增，核心） `[IMPLEMENTED]`

### 3.1 RuntimeManifest 结构（domain，零外部依赖）

放 `contexts/agent-core/domain/runtime-manifest.ts`。固化字段与替换的运行时调用：

| 字段 | 固化自 | 替换的运行时调用 |
|---|---|---|
| `compiledSystemPrompt` | `persona.systemPrompt` | `personaProvider.getPersona`（harness.ts:228）的 systemPrompt 分支 |
| `compiledGuardrails` | `persona.guardrails`（GuardrailRule[]） | `getPersona` 的 guardrails 分支 |
| `compiledTools` | `boundTools` 经 assembleTools 解析为 `[{name,description,inputSchema}]` | `assemblyProvider.assemble`（harness.ts:191）的 tools 部分 |
| `compiledSkillsContext` | `boundSkills` 经 assembleSkills 拼接 | `assemble` 的 skillsContext 部分 |
| `compiledQuota` | `resourceLimits` + `modelConfig` 快照 | 运行时配额检查 |
| `runtimeRoute` | `runtime.runtimeType` → adapter 映射 | `selectBestAdapter`（adapter-registry.ts:60）路由 |
| `sandboxStrategy` | `sandboxTemplate` → executor（opensandbox/cubesandbox） | executor 选择 |

Manifest 含元信息：`id` / `agentDefinitionId` / `generation` / `bakedAt` / `status`。

### 3.2 不可变性保证

- domain 层 `sealManifest(manifest)` 函数：bake 完成时调用，返回 frozen 不可变对象（`Object.freeze` 递归）。
- DB 层：唯一约束（agentDefinitionId + generation）——同一定义同一代际只一个 manifest；`status='baked'` 后 repository 拒绝 update manifest 字段（只允许 status 状态流转）。
- 改 spec → `bumpGeneration`（agent-definition.ts:251）→ re-bake 新 generation 的 manifest；旧 instance 引用旧 generation manifest（灰度/回滚天然支持）。

### 3.3 BakingService（application，编排 bake）

放 `contexts/agent-core/application/baking-service.ts`。

**bake 流程**：
```
bake(agentDefinitionId, generation):
  1. 取 AgentDefinition(spec) — 经 IAgentDefinitionPort.getById(已有,assembly-provider.ts:18)
  2. 落 manifest 记录 status=pending(占位,防并发重复 bake)
  3. 同步:复用 assembleTools/assembleSkills(提纯后,C4)解析 boundTools/boundSkills
     + 取 persona.systemPrompt/guardrails
     + 快照 quota + 计算 runtimeRoute/sandboxStrategy
  4. sealManifest(固化) → 落 DB manifest jsonb + status=baked
  5. 任一步失败 → status=failed + errorMsg(不污染 baked manifest)
```

**同步固化**：bake 同步执行 assembleTools/assembleSkills + persona 快照 + seal，返回终态（baked|failed）。bake 是低频管理操作且均为本地 DB 查询（百毫秒级），同步返回终态比异步轮询简单，且无 pending 卡死风险（server 重启 pending 永卡）。规模大时再迁 scheduler 异步队列（远期方向不变）。

**复用而非重发明**（§9.7）：`assembleTools`（assembly-provider.ts:187）/`assembleSkills`（:260）当前是 AssemblyProvider 实例方法。C4 提为可复用纯函数（或独立 module），bake 时调、运行时不再调。解析逻辑（跨租户校验/禁用跳过/inputSchema 缺省降级/空数组语义陷阱）只有一份实现。

### 3.4 DB schema

`agent_runtime_manifests` 表：
| 列 | 类型 | 说明 |
|---|---|---|
| id | text pk | manifest id（newId('rman')） |
| agentDefinitionId | text | 外键 agent_definitions |
| generation | integer | spec 代际 |
| manifest | jsonb | 固化产物 |
| status | text | pending/baked/failed/expired |
| bakedAt | timestamptz | 固化完成时间 |
| errorMsg | text | failed 原因 |

- **唯一约束**：`(agentDefinitionId, generation)`
- **索引**：`agentDefinitionId`（单列，查某定义全部 manifest）、`status`（监控 baked/pending/failed）
- migration 必须 `.ts`（§7.2.1 第3条；记忆 `migrate.ts 不跑 .sql`，tsc/vitest 测不出缺列只有真请求暴露）
- repository 放 `db/repositories/runtime-manifest-repository.ts`（仿 sandbox-session-repository 模式）

## 4. CubeSandbox 实证（推翻 T53 旧判断） `[PLANNED]`

T53 记忆否决 CubeSandbox 的理由"无 JS SDK + 需 KVM 定制内核"——读 README 实证后**部分推翻**：

| T53 旧判断 | 实证（2026-06） | 结论 |
|---|---|---|
| 无 JS SDK | 原生兼容 E2B REST API + E2B 官方 Node SDK（`e2b` 包），换 URL 环境变量即接入 | ✅ 推翻，Node 栈可用 |
| 需 KVM 定制内核 | 需支持 KVM 的 x86_64 Linux，PVM 云服务器可（无需裸金属/嵌套虚拟化），无需定制内核 | ⚠️ KVM 仍是硬要求（T53 对一半） |

**CubeSandbox vs OpenSandbox 安全等级梯度**（并存定位）：

| 维度 | OpenSandbox | CubeSandbox |
|---|---|---|
| 隔离 | docker 共享内核 | KVM MicroVM 独立内核 + eBPF 网络隔离 |
| 启动 | ~200ms | <60ms 冷启动 |
| 部署 | docker 一行，无 KVM | 需 KVM 宿主 |
| 场景 | 默认/开发/轻量 | 不可信代码/多租户高密度/高安全 |

**决策**：并存非替换。OpenSandbox 默认，CubeSandbox 作高安全可选执行层（sandboxTemplate='kvm-microvm' 路由）。

## 5. Layer 3 运行时动态组装层（改造，灰度） `[IMPLEMENTED]`

### 5.1 IRuntimeManifestPort

放 `contexts/agent-core/domain/runtime-manifest-port.ts`，仿 `IInstanceLookupPort`（assembly-provider.ts:13）模式（消费方定义 port，守 §1.3 跨聚合纪律，T47 同款）。

```ts
interface IRuntimeManifestPort {
  getManifest(instanceId: string): Promise<RuntimeManifest | null>;
  // instance → agentDefinitionId + generation → 查 manifest
}
```

bootstrap 适配注入（仿 adaptInstanceLookup）。

### 5.2 harness 改造（harness.ts:179-244）

```
dispatchTask:
  manifest = runtimeManifestPort.getManifest(instanceId)  // 优先
  if (manifest?.status === 'baked'):
    input.allowedTools      = manifest.compiledTools.map(name)
    input.externalTools     = manifest.compiledTools   // worker 路径
    input.skillsContext     = manifest.compiledSkillsContext
    input.systemPrompt      = manifest.compiledSystemPrompt
    guardrails              = manifest.compiledGuardrails  // 拦截用
    // 不再调 assemble + getPersona(声明态产物已固化)
  else:
    // 降级老路径(灰度兼容存量未 bake 的 Agent)
    assemble(...) + getPersona(...)
  
  // 运行时仍动态组装(不从 manifest 读):
  //   凭证解密 / session / 实时配额检查(quota 当前余量) / 用户输入 / trace
```

**灰度策略**：有 baked manifest 走 manifest，无则降级老路径。assembly-provider / personaProvider **保留**作降级，不废弃（bake 普及后单独收敛任务，不在 v2.0 强行收敛，避免半新半旧违反 §4.3）。

### 5.3 运行分离：CubeSandbox 并存

#### CubeSandboxExecutor

放 `contexts/tool-management/executors/cubesandbox-executor.ts`，平行 `OpenSandboxExecutor`：
- `implements IToolExecutor`，`execute(config, params, credential)` 按 op（write_file/read_file/list_files）分流——接口契约与 OpenSandboxExecutor 一致（opensandbox-executor.ts:13 注释自陈"上层零改动"）。
- 底层 E2B Node SDK（`e2b` 包）：`Sandbox.create/connect` + `files.read/write` + `process` + 端口转发，端点指向 CubeSandbox CubeAPI 网关。
- 复用 OpenSandboxExecutor 的：`getOrCreateSandbox` 三段式缓存+持久化（进程缓存→DB sandboxId 重连→新建）、`mapSearchEntries` 纯函数、`resolveWorkspacePath` 路径校验。
- 沙箱 session 复用现有 `sandbox_sessions` 表（T55 已落地，callerId↔sandboxId），C10 加 `backend` 字段区分 opensandbox/cubesandbox。

#### SandboxRouter

放 `contexts/tool-management/executors/sandbox-router.ts`：
- 当前 executor-factory.ts:23 sandbox 是模块单例二选一（OPENSANDBOX_DOMAIN ? OpenSandbox : node-fs）。并存需路由层。
- SandboxRouter `implements IToolExecutor`，按 manifest.sandboxStrategy（或 tool config.sandboxBackend）路由：`'kvm-microvm'` → CubeSandboxExecutor，其余 → OpenSandboxExecutor。
- `getExecutor('sandbox')` 返回 SandboxRouter（内部分发）。
- sandboxStrategy 来源：通过 `params.__callerId` → instance → manifest.sandboxStrategy（与现有 __callerId 注入机制一致）；或 tool config 显式声明 `sandboxBackend`（bake 时固化进 tool 定义，更简单）。

#### config

`config/index.ts` 加 cubesandbox 配置块（domain/apiKey/image）+ `.env.example`（CUBESANDBOX_DOMAIN/CUBESANDBOX_API_KEY）。bootstrap 装配 CubeSandboxExecutor + SandboxRouter + BakingService + RuntimeManifestPort 适配。

## 8. 风险与约束 `[IMPLEMENTED]`

- **灰度兼容**：harness 必须保留 assemble+getPersona 降级路径，否则未 bake 的存量 Agent 全挂。C6 是灰度关键，不可跳过降级分支。
- **bake 一致性**：spec 变更到 manifest baked 之间的窗口期，instance 可能读到旧 manifest（可接受，generation 隔离）。但需防"读到 pending manifest"——harness 只认 status=baked，pending/failed 走降级。
- **CubeSandbox KVM 依赖**：代码只接 E2B SDK，KVM 宿主部署是运维侧。无 KVM 环境时 CubeSandboxExecutor 不可用，SandboxRouter 降级 OpenSandbox（配置缺失即不启用，仿 OPENSANDBOX_DOMAIN 模式）。
- **复用边界**：C4 提纯 assembleTools/assembleSkills 必须保证解析行为不变（跨租户校验/禁用跳过/空数组陷阱），有现成测试守护（assembly-provider.test.ts）。
- **DB 迁移**：migrate.ts 不跑 .sql（记忆），C2/C10 必须 .ts migration，否则运行时缺列 500（tsc/vitest 测不出）。

## 10. 对后续版本的约束 `[IMPLEMENTED]`

1. **manifest 成为运行时真相源**：bake 普及后，assembly-provider/personaProvider 可收敛删除（单独版本任务），届时 harness 只读 manifest。
2. **sandbox 多后端路由固化**：SandboxRouter 模式确立后，新增沙箱后端（Firecracker/gVisor 等）只加 executor + router 分支，不动上层。
3. **Agent 行为不可变契约**：baked manifest 不可变，线上 Agent 行为可锁定可审计可回滚——这是商业模式可信（v1.2.2 计费）的前置：计费基于固化配额才不漂移。
4. **版本号纪律**：snapshot 不可改，版本号不重复用（v1.3 已归档占用，故升 v2.0）。
