# 15 — 部署模型与构建策略

## 部署模型

企业级产品，数据敏感度极高。部署模型直接决定能不能进客户的门。

| 模式 | 适用场景 | 核心考量 |
|------|---------|---------|
| 私有化部署 | 大企业、金融、政府 | 数据不出企业边界，安全合规优先 |
| 专有云 | 中大型企业 | 平衡安全与运维成本 |
| 混合模式 | 多数场景的现实选择 | 敏感数据在内、Agent 运行时在外 |

### 底线

**核心运行时和数据层必须支持私有化部署**——这是进入企业市场的门票。

Agent 运行层可以弹性部署在云端（因为 LLM API 本来就在云上）。

## 构建策略：自建 vs 复用

不要什么都自己造。

| 组件 | 策略 | 原因 |
|------|------|------|
| 连接器（数据接入） | 复用开源 + 自建适配 | 脏活累活，不是核心壁垒 |
| 事件流 | 复用（Kafka / Pulsar） | 成熟基础设施，没必要重造 |
| Agent 运行时 | 自建抽象层 + 适配各框架 | **核心差异化，必须自己控制** |
| 编排引擎 | 基于 Temporal 等二次开发 | 底座复用，组织级逻辑自建 |
| 信号总线 | 自建 | **核心智能，不能外依赖** |
| 应用服务层 | 全部自建 | **业务逻辑是核心价值** |
| 用户界面 | 自建 | 直接决定用户体验和付费意愿 |

### 关键判断

真正需要自建的是三块：**Agent 抽象层、信号总线、应用服务层**。其余能复用就复用，把精力集中在核心差异化上。

## 产品形态总结

最接近的类比：**Kubernetes + Datadog + Jira 的三合一体**

| 类比 | 对应能力 | 本系统的差异 |
|------|---------|-------------|
| Kubernetes | 容器编排 → Agent 编排 | 编排对象从容器变为 Agent，调度逻辑从资源驱动变为目标驱动 |
| Datadog | 基础设施可观测 → 组织可感知 | 监控对象从服务器/服务变为 Agent 执行和组织信号 |
| Jira | 人的任务管理 → 人机混合任务管理 | 任务承载者从纯人变为人 + Agent，评价标准分轨 |

但服务的对象从"容器和开发者"变成了"Agent 和组织决策者"。
