## 子系统一：战略解码引擎（Strategy Decomposition Engine）

### 职责

老板输入模糊的自然语言级别的战略意图（如"明年东南亚市场营收翻倍"），引擎负责：

- 将战略意图拆解为**可执行的任务树**
- 识别任务间的依赖关系、优先级、资源约束
- 拆解结果不是终态，是初始假设，需要持续被修正

### 与传统项目管理的区别

Jira 拆的是人的任务，这个拆的是 Agent 可执行的任务单元。粒度、结构、验收标准完全不同。

---

## 子系统二：智能体编排层（Agent Orchestration Layer）

### 职责——OS 的内核

类比 Kubernetes 调度容器，这层调度 Agent：

| 能力 | 说明 |
|------|------|
| Agent 注册与能力描述 | 每个 Agent 能干什么、擅长什么、成本多少 |
| 任务路由与分配 | 哪个任务给哪个 Agent，并行还是串行 |
| 上下文管理 | Agent 间的信息传递、共享记忆、状态同步 |
| 异常处理 | Agent 失败了怎么重试、降级、升级给人 |

### 当前市场状态

CrewAI、AutoGen、LangGraph 都在做这层的雏形，但全是技术视角，缺乏组织视角。

---

## 子系统三：人机判断界面（Human Judgment Interface）

### 职责

"少数关键人才"的工作台。不是传统 Dashboard，而是：

- **信号聚合视图**：把 Agent 执行中产生的异常信号、模式变化、关键决策点聚合呈现
- **判断节点**：系统识别"这个决策超出 Agent 能力范围"，主动推给人
- **纠偏操作**：人做出判断后，系统自动将修正传播到所有受影响的下游 Agent 和任务

### 核心设计原则

**人不应该去找问题，问题应该来找人。**

---

## 子系统四：感知与反馈回路（Sensing & Feedback Loop）

### 职责

解决"老板变瞎子"问题的关键：

- 从 Agent 执行数据中提取**涌现信号**（不是 KPI 报表，是"好像有什么不对劲"的弱信号）
- 跨 Agent 的**模式识别**（三个不同方向的 Agent 都遇到类似阻力，意味着什么？）
- 自动生成**战略假设验证报告**（当初拆解时的假设验证了多少？偏了多少？）

### 定位

整个系统的神经系统。没有它，上面三层全是断裂的。

---

## 子系统五：组织知识沉淀层（Organizational Knowledge Layer）

### 职责

Agent 执行完一轮，组织学到了什么？

- 不是把数据存下来，是把**决策模式和判断逻辑**沉淀下来
- 下一轮战略拆解时，系统能基于历史经验给出更准确的初始方案
- 人才离职时，他的判断经验不完全流失
