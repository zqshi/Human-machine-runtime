# 06 — Agent 可插拔架构与任务契约

## 核心判断

Agent 框架处于快速迭代期，绑死任何一个都是找死。系统架构必须保证 Agent 层**可插拔**。

## 任务契约（Task Contract）

战略解码引擎下发的不是"给某个特定 Agent 框架的指令"，而是一个标准化的任务契约：

```
┌───────────────────────────────────┐
│  任务契约（Task Contract）         │
│  ├── 目标：要达成什么              │
│  ├── 输入：给什么上下文            │
│  ├── 验收标准：怎么算完成          │
│  ├── 约束：不能碰什么、预算多少     │
│  └── 上报条件：什么情况要报告人     │
└───────────────────────────────────┘
```

具体由哪个 Agent 框架执行（OpenClaw / Harness / 自研），由编排层根据以下因素动态路由：

- 任务类型匹配度
- Agent 能力评分（基于历史表现）
- 执行成本
- 历史成功率

甚至同一个任务可以让两个 Agent 并行执行再比对结果。

## 抽象层的意义

这层抽象做好了 → Agent 框架随便换，系统不受影响。

这层抽象做不好 → 换一个框架整个系统要重写。
