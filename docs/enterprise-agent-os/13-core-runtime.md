# 13 — 核心运行时：OS 的内核

## 定位

整套系统最重、最难、也最有价值的部分。由三大组件构成。

---

## 组件一：编排引擎（Orchestration Engine）

### 不是传统工作流引擎

它要处理的是**动态 DAG**——任务图在执行过程中会变：

```
传统工作流引擎：A → B → C（定义好了不变）
编排引擎要做的：A → B → C（执行到 B 发现需要插入 D，且 C 的验收标准变了）
```

### 核心能力

- 动态任务图管理（运行时增删改节点和边）
- 基于 Agent 能力评分的动态路由
- 异常处理与升级（失败重试 → 降级方案 → 升级给人）
- 并发控制与资源竞争管理

### 技术选型方向

Temporal / Prefect 这类支持动态工作流的引擎作为底座，上面封装组织级编排逻辑。

### 构建策略

基于 Temporal 等二次开发。底座复用，组织级逻辑自建。

---

## 组件二：Agent 抽象层（Agent Abstraction Layer）

### 核心职责

为编排引擎和 Agent 运行层之间提供统一接口：

- Agent 注册与能力描述（每个 Agent 能干什么、擅长什么、成本多少）
- 任务契约标准（统一的输入/输出/验收格式）
- Agent 生命周期管理（启动、暂停、终止、替换）
- 能力评分与路由权重更新

### 设计原则

编排引擎只与抽象层交互，永远不直接调用具体的 Agent 框架。这保证了 Agent 可插拔。

---

## 组件三：信号总线（Signal Bus）

### 不只是消息队列

信号总线是所有子系统之间的通信中枢，要做**信号级别的处理**：

| 能力 | 说明 |
|------|------|
| 噪声过滤 | 从海量事件中识别有意义的信号 |
| 跨 Agent 信号关联 | 三个 Agent 都报了类似异常 → 聚合为一个系统级信号 |
| 信号分级与路由 | 普通信号存档，重要信号推给人 |
| 模式检测 | 识别反复出现的异常模式，预警潜在系统性问题 |

### 定位

整个系统的**神经系统**。没有它，编排引擎、应用服务、用户界面全是断裂的信息孤岛。

### 构建策略

**必须自建。** 信号检测和模式识别是核心智能，不能依赖外部。
