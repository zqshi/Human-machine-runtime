## 1. 功能概述

### 1.1 功能定位

线上回收与待审用例是评测中心的**基准自增长**机制。当数字员工在线上运行中出现任务失败、用户纠正、安全违规等事件时，系统自动将这些失败案例回收为候选评测用例，进入待审队列。人工审核后，可将高质量候选用例提升为正式评测用例，纳入评测集。这解决了评测基准"手工构建慢、覆盖不全"的问题——线上真实失败是最有价值的测试用例来源。

### 1.2 目标用户

| 用户类型 | 特征描述 | 核心诉求 |
|---------|---------|---------|
| 评测工程师 | 负责构建和维护评测基准 | 快速将线上失败转化为评测用例，减少手工编写 |
| 运维工程师 | 负责线上质量监控 | 查看待审用例了解线上失败分布，决定是否采纳 |

### 1.3 核心价值

| 维度 | 描述 |
|------|------|
| 用户价值 | 评测基准自动增长，无需人工逐条编写用例，覆盖率持续提升 |
| 业务价值 | 线上失败不再"发生即遗忘"，而是沉淀为回归测试资产，防止同类问题再次上线 |
| 技术价值 | 闭环链路：线上 Trace → 自动采集 → 待审队列 → 人工审核 → 提升为 EvalCase |

### 2.3 详细规格

#### F-R01：待审用例面板

**功能描述**: 在实验中心看板的统计卡片区，"待审 Case"卡片可点击展开待审用例面板。面板展示所有 pending 状态的候选用例，包含完整信息供审核人判断。

**前置条件**:
- 用户在实验中心页面
- 待审 Case 数 > 0（统计卡片橙色高亮）

**主要行为**:
1. 点击「待审 Case」统计卡片，展开待审用例面板
2. 面板顶部支持筛选：按触发原因、按失败模式
3. 每条候选用例展示以下信息：
   - **原始用户输入**（originalInput）：用户对 Agent 说了什么
   - **Agent 输出**（agentOutput）：Agent 实际返回了什么
   - **用户纠正**（userCorrection）：用户是否提供了正确答案（如有）
   - **触发原因**（triggerReason）：为何被回收
   - **失败模式**（failureMode）：失败的具体分类
   - **来源 Trace ID**：可点击跳转到追踪详情
   - **回收时间**（createdAt）
4. 点击「采纳」进入提升流程（见 F-R02）
5. 点击「忽略」标记为 rejected（见 F-R03）

**业务规则**:
- 默认只展示 pending 状态的用例
- 面板最大高度 300px，超出滚动
- 触发原因和失败模式支持筛选标签

**边界条件**:
| 边界场景 | 处理方式 |
|---------|---------|
| 无待审用例 | 面板展示"没有待审用例" |
| 加载中 | 展示加载中占位 |
| Agent 输出过长 | 截断展示，点击展开完整内容 |

#### F-R02：采纳并提升

**功能描述**: 采纳候选用例并提升为指定评测集下的正式用例。采纳和提升是两步操作：先审核通过（approved），再指定目标评测集提升（promoted）。

**前置条件**:
- 用户在待审用例面板点击「采纳」

**主要行为**:
1. 点击「采纳」后，弹出提升配置弹窗
2. 选择**目标评测集**（必选，下拉列表）
3. 确认**用户输入**（默认取 originalInput，可编辑）
4. 确认**期望行为**（默认取 userCorrection，如无则需手动填写）
5. 系统自动填充以下默认值：
   - caseKey：`REPLAY-{id}`
   - category：`线上回收`
   - evalType：`behavioral`
   - regressionSource：来源 traceId
6. 点击「确认提升」
7. 用例创建到目标评测集，状态为 active
8. 候选用例状态变为 promoted，记录 promotedCaseId

**业务规则**:
- 目标评测集为必选，若无可用的评测集，提示"请先创建评测集"
- 提升后候选用例状态从 pending → promoted（不是 approved）
- promoted 状态表示已成功提升为正式用例，与 approved（仅审核通过但未提升）不同
- 提升后的用例可在目标评测集详情页查看和编辑
- 同一条候选用例只能提升一次

**边界条件**:
| 边界场景 | 处理方式 |
|---------|---------|
| 无可用评测集 | 提示"请先创建评测集"，采纳按钮禁用 |
| 用户纠正为空 | 期望行为字段为空，需用户手动填写 |
| 提升失败 | 保持 pending 状态，提示错误信息 |

---

#### F-R03：忽略候选用例

**功能描述**: 将不需要的候选用例标记为 rejected，从待审列表移除。

**前置条件**:
- 用户在待审用例面板点击「忽略」

**主要行为**:
1. 点击「忽略」
2. 候选用例状态变为 rejected
3. 从待审列表移除（不再展示在 pending 视图）
4. 记录审核人和审核时间

**业务规则**:
- 忽略操作无需二次确认
- 忽略后用例仍可通过审核历史查看（F-R04）
- 忽略后不可恢复（如需恢复，需在审核历史中操作）

---

#### F-R04：审核历史查看

**功能描述**: 支持查看已审核的用例（approved / rejected / promoted），供回溯和审计。

**前置条件**:
- 用户在待审用例面板切换到"已审核"视图

**主要行为**:
1. 面板顶部增加视图切换：待审 / 已审核
2. 已审核视图展示 approved / rejected / promoted 状态的用例
3. promoted 状态的用例展示关联的评测集和 Case ID（可点击跳转）

**业务规则**:
- 已审核视图按审核时间倒序
- promoted 状态的用例展示 promotedCaseId 链接

#### F-R05：线上自动采集

**功能描述**: 当数字员工在线上运行中出现特定事件时，系统自动将相关 Trace 信息写入 Replay 队列。

**前置条件**:
- 数字员工在线上环境运行
- Trace 系统已接入

**采集触发条件**:

| 触发原因 | 触发条件 | 采集内容 | 说明 |
|---------|---------|---------|------|
| task_failed | Agent 执行返回错误或超时 | originalInput + agentOutput | 任务执行失败 |
| human_correction | 用户对 Agent 输出进行了纠正 | originalInput + agentOutput + userCorrection | 用户提供了正确答案，最有价值的回收来源 |
| user_abandoned | 用户在 Agent 响应后立即退出/重试 | originalInput + agentOutput | 用户不满意但未纠正 |
| safety_violation | 安全检查拦截了 Agent 操作 | originalInput + agentOutput | 安全边界违规 |
| high_cost | 单次执行 Token 消耗超过阈值 | originalInput + agentOutput | 成本异常 |
| low_score | 评测实验中用例得分低于 0.3 | originalInput + agentOutput | 低分用例自动回收 |

**失败模式分类**:

| 失败模式 | 说明 |
|---------|------|
| tool_error | 工具调用失败（网络错误、参数错误等） |
| wrong_output | 输出内容错误（事实错误、逻辑错误） |
| incomplete | 输出不完整（遗漏关键信息） |
| format_error | 输出格式错误（非预期 JSON 结构） |
| safety_breach | 违反安全约束（越权、注入等） |
| timeout | 执行超时 |
| other | 其他未分类 |

**业务规则**:
- 采集由后端 Trace 监听服务异步写入，不阻塞 Agent 执行
- 同一 Trace ID 不重复写入（幂等）
- 采集后自动进入 pending 状态
- 采集频率限制：同一数字员工同一小时内最多回收 100 条，超出丢弃

## 3. 数据模型

### EvalReplayQueue（线上回收队列）

| 字段名 | 类型 | 必填 | 说明 | 约束 |
|--------|------|------|------|------|
| id | serial | 是 | 自增主键 | — |
| traceId | string | 是 | 来源 Trace ID | 最大 128 字符，唯一 |
| triggerReason | enum | 是 | 触发原因 | task_failed / human_correction / user_abandoned / safety_violation / high_cost / low_score |
| originalInput | text | 否 | 原始用户输入 | — |
| agentOutput | text | 否 | Agent 输出 | — |
| userCorrection | text | 否 | 用户纠正后的输出 | 仅 human_correction 类型有值 |
| failureMode | enum | 否 | 失败模式分类 | tool_error / wrong_output / incomplete / format_error / safety_breach / timeout / other |
| reviewStatus | enum | 是 | 审核状态 | pending / approved / rejected / promoted |
| promotedCaseId | string | 否 | 提升后对应的 Case ID | 仅 promoted 状态有值 |
| reviewedBy | string | 否 | 审核人 | — |
| reviewedAt | timestamp | 否 | 审核时间 | — |
| tenantId | string | 否 | 租户 ID | — |
| createdAt | timestamp | 是 | 创建时间 | — |

### 审核状态流转

```
pending --[采纳]--> promoted（提升为正式用例）
pending --[忽略]--> rejected
pending --[审核通过]--> approved（仅审核，未提升）
approved --[提升]--> promoted
```

**状态说明**:
| 状态 | 含义 |
|------|------|
| pending | 待审核，刚从线上回收 |
| approved | 人工审核通过，但尚未提升到评测集 |
| rejected | 人工忽略，不采纳 |
| promoted | 已提升为正式评测用例，关联 promotedCaseId |

## 4. 交互流程

### 4.1 主流程 — 采纳并提升

```
[起始点] 用户在实验中心查看待审用例
    ↓
[步骤1] 点击某条候选用例，查看 Agent 输出和用户纠正
    ↓
[判断] 是否值得采纳？
    ↓ [是]                    ↓ [否]
[步骤2a] 点击「采纳」       [步骤2b] 点击「忽略」
    ↓                         ↓
[步骤3] 弹出提升配置弹窗    [结束] 状态变为 rejected
    ↓
[步骤4] 选择目标评测集
    ↓
[步骤5] 确认用户输入和期望行为
    ↓
[步骤6] 点击「确认提升」
    ↓
[步骤7] 用例创建到目标评测集，状态变为 promoted
    ↓
[结束] 候选用例从待审列表移除
```

### 4.2 异常分支

| 异常场景 | 触发条件 | 处理方式 | 用户提示 |
|---------|---------|---------|---------|
| 无可用评测集 | 系统中无评测集 | 采纳按钮禁用 | "请先创建评测集" |
| 提升失败 | 后端创建用例异常 | 保持 pending 状态 | "提升失败，请重试" |
| 重复提升 | 同一候选用例已 promoted | 忽略操作 | — |
| 采集限流 | 同一员工同一小时超 100 条 | 超出丢弃 | — |
