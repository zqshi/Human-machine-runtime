## 1. 功能概述

### 1.1 功能定位

定时任务管理是 HMR 组织控制面下的**自动化子系统**，位于管理后台「自动化 → 定时任务」。它把"周期性、可推迟、需留痕"的运维与运营动作，从"人记得去做"转变为"系统到点自动执行"。用户在面板上配置任务（做什么 + 多久做一次），系统到点自动执行、留存每次执行的结论与产出报告，并可在列表与详情页全程回溯。

它同时是平台所有自动化作业的**统一入口**：既包含面向数字员工的"定时执行一次任务"（agent 类型），也包含平台自身的系统作业（system 类型：链路自检、数据清理、离职监测、周期报告）。**本期重点**是把分散的定时任务在管理后台收口、统一呈现，让运营及相关方能在一处查看全部任务的运行状态、执行节奏与产出结论。

### 1.2 目标用户

| 用户类型 | 特征描述 | 核心诉求 |
|---------|---------|---------|
| 平台管理员 | 负责平台稳定性与数据合规，关心成本、留存、数据堆积 | 自动清理过期数据、监测离职员工、周期产出运营报告，少人工值守 |
| 数字员工运营者 | 管理数字员工实例，希望员工定时干活 | 让某个数字员工按固定周期自动执行一次指令并拿回结论 |

> "开发/接入方"低成本注册新作业类型的能力作为**后续版本**支持（见附录 C-Q05）。

### 1.3 核心价值

| 维度 | 描述 |
|------|------|
| 用户价值 | 周期性动作"配置即生效"，无需写代码、无需值守；每次执行结论可查，问题可回溯 |
| 业务价值 | 数据生命周期、账号生命周期、运营洞察全部自动化闭环，降低合规与人力成本 |
| 技术价值 | 统一的 DB-tick 调度 + 分布式锁 + handler 注册表，调度/执行/模板解耦，为后续低成本扩展作业类型预留架构 |

### 2.3 详细规格

#### F-01：任务列表与筛选

**功能描述**: 进入「自动化 → 定时任务」展示当前组织下所有定时任务，提供统计概览、筛选与搜索，每行提供快捷操作。

**前置条件**:
- 用户已登录管理后台并具备定时任务访问权限
- 已通过列表接口拉取任务

**主要行为**:
1. 顶部展示标题与副标题「管理周期性自动执行的任务，查看每次执行与产出结论」，右侧「新建任务」按钮
2. 统计卡区展示三项：**任务总数**、**启用中**（绿色）、**最近异常**（红色，统计上次结果为 failed/timeout 的任务数）
3. 筛选条：`全部 / 启用 / 暂停` 三态切换；右侧任务名搜索框
4. 任务列表逐行展示：任务名（可点击进详情）、类型标签（数字员工/系统作业）、暂停标签、上次结果（成功绿/失败红/超时橙）、下次执行倒计时（仅启用任务）、调度表达式（cron 等宽显示 或 `每 Ns`）、最近错误摘要（截断 50 字符）
5. 每行右侧操作区：手动执行、查看详情与历史、启用/暂停、编辑、删除

**业务规则**:
- 「下次执行」倒计时每 60 秒自动刷新（列表与详情页均刷新）；显示为"即将执行 / N 分钟后 / N 小时后 / N 天后 / 已到期"；无调度计划显示"未调度"
- 仅 `启用中` 的任务展示倒计时；已暂停任务展示"已暂停"标签
- 搜索为任务名模糊匹配（大小写不敏感）
- 「手动执行」按钮在请求期间禁用并切换为沙漏图标，防重复点击

**边界条件**:
| 边界场景 | 处理方式 |
|---------|---------|
| 加载中 | 显示 3 行骨架屏（脉冲动画） |
| 加载失败 | 红色错误文案 + 「重试」按钮 |
| 无任务（空状态） | 空状态页："暂无定时任务"，引导「新建任务」 |
| 搜索无匹配 | 空状态页："没有匹配的任务，换个关键词试试" |
| 启用筛选与搜索叠加 | 先按启用状态过滤，再按名称过滤 |

#### F-02：任务模板与动态配置表单

**功能描述**: 新建/编辑任务以右侧抽屉形式打开，用户先选择"任务模板"，表单按所选模板动态渲染配置项，支持字段联动（条件显示）。

**前置条件**:
- 用户点击「新建任务」或在列表/详情点击「编辑」
- 已加载任务模板注册表

**主要行为**:
1. 抽屉标题为「新建定时任务」或「编辑定时任务」
2. 依次填写：任务名称（必填）、描述（可选）
3. 「任务模板」区以 2 列卡片网格展示所有模板，卡片含图标、名称、一句话描述；选中高亮
4. 切换模板时，动态表单按新模板的字段集重新渲染并重置为默认值
5. 表单字段按模板定义的 `showWhen` 条件联动显示（如勾选某维度才出现对应参数）
6. 调度频次（见 F-03）
7. 「立即启用」复选框（默认勾选）
8. 底部「取消」「保存」

**业务规则**:
- 任务名必填，为空时提示"请填写任务名称"
- 各模板的必填字段（`required`）为空时提示"请填写「{字段名}」"
- 存在 JSON 格式错误时禁止保存，提示"存在 JSON 格式错误"
- 若所选模板标记为执行能力未就绪（`backendPending`），顶部橙色提示条："⚠ 该作业的后端 handler 尚未实现，可保存配置，但调度时不会真正执行"
- 仅提交"当前可见字段"的值（被联动隐藏的字段不写入），并按模板注入 `handlerKey` 等内部字段
- 编辑模式以 `key=任务id` 控制组件重建，确保表单状态与目标任务一致

**字段类型支持**（动态表单渲染器）:

| 类型 | 渲染形态 | 值约定 |
|------|---------|--------|
| text | 单行输入 | string |
| number | 数字输入 | number |
| textarea | 多行输入 | string |
| markdown | 富文本编辑器（工具栏 + 编辑/预览切换） | string |
| json | JSON 文本域（实时校验） | object |
| select | 下拉选择 | string |
| checkbox-group | 多选按钮组（可按 group 分组） | string[] |
| tag-input | 标签输入（回车/逗号追加，退格删除） | string[] |
| boolean | 复选框 | boolean |

#### F-03：调度频次配置

**功能描述**: 以"人类友好"方式配置执行频次，底层生成调度表达式；自定义 cron 时提供实时校验与触发预览。

**前置条件**:
- 已选择任务模板（部分模板会收窄可选频次）

**主要行为**:
1. 频次模式 5 选 1 按钮：**固定间隔 / 每天 / 每周 / 每月 / 自定义**
2. 固定间隔：输入"每 N 分钟/小时 执行一次"
3. 每天：选时间点（HH:MM）
4. 每周：选星期（周日~周六）+ 时间点
5. 每月：选每月几号（1~31）+ 时间点
6. 自定义：Cron 表达式输入框，400ms 防抖后实时校验
7. 底部等宽预览生成的 `cron` 或 `间隔` 表达式

**业务规则**:
- 频次模式与底层表达式的相互反推保持一致（编辑旧任务时正确还原模式）
- 模板可声明 `allowedFreqModes` 限制可选频次：**报告统计模板仅允许"每周/每月"**，因为其统计窗口由频次自动推导（每周→上周，每月→近 30 天），避免频次与统计范围冲突；编辑时若旧任务频次不在允许范围内，回落到允许的第一个
- 报告统计模板下额外提示："统计窗口由频次自动推导：每周 → 上周，每月 → 近 30 天"
- Cron 校验返回：是否合法 + 人类可读描述（如"每周一 9:00"）+ 接下来 3 次触发时间预览；非法时显示红色错误
- Cron 时区默认 `Asia/Shanghai`

**边界条件**:
| 边界场景 | 处理方式 |
|---------|---------|
| Cron 输入为空 | 不校验，不展示预览 |
| Cron 格式非法 | 红色"✗ {错误}"，禁止保存 |
| Cron 合法 | 绿色"✓ {描述}" + 接下来 3 次触发时间 |
| 校验请求失败 | 静默不阻塞（不展示校验结果） |
| 每月填入 >31 的号 | 钳制到 31 |
| 间隔填入 <1 | 钳制到 1 |

#### F-04：启用/暂停与手动执行

**功能描述**: 在列表行操作和详情页对单个任务进行启用/暂停切换、手动触发一次执行。

**前置条件**:
- 任务已存在

**主要行为**:
1. **启用/暂停**：切换 `is_enabled`，立即生效；暂停后不再被调度，但保留全部配置与历史；启用后重算下次执行时间
2. **手动执行**：立即触发一次，产出一条 `manual` 触发的执行记录；详情页执行后自动刷新历史

**业务规则**:
- **手动触发不改变既有调度节奏**：仅 `scheduled`（定时）触发才会推进 `nextRunAt`；`manual` 触发执行完毕后，下次执行时间仍按原计划
- 多副本部署下，同一任务同一时刻由分布式锁保证只执行一次
- 单进程内长任务执行期间，下一次 tick 不会重入同一任务（in-flight 互斥）
- 单次执行默认超时上限 5 分钟，超时记为 `timeout`
- 任务删除需二次确认，且级联删除其全部执行历史

**边界条件**:
| 边界场景 | 处理方式 |
|---------|---------|
| 暂停中的任务手动执行 | 允许（手动执行不依赖启用状态） |
| 同一任务被另一副本持锁 | 当前副本跳过，不重复执行 |
| 执行超时（>5min） | 记录 `timeout` 状态与错误信息 |
| 执行抛错 | 记录 `failed` 状态与错误信息，任务仍可被下次调度 |
| 删除任务 | 二次确认（含任务名），确认后删除任务 + 级联删除执行记录 |

#### F-05：详情页与执行历史

**功能描述**: 列表点击任务名进入整页详情，展示任务元信息、执行时间与统计、双栏的执行历史与输出报告。

**前置条件**:
- 用户在列表点击任务行或「查看详情」

**主要行为**:
1. 顶部：返回列表、任务名、类型标签、启用/暂停标签、描述；右侧「立即执行」「编辑」
2. 第一行统计卡：调度频次、下次执行、上次执行、上次结果（失败时红色）
3. 第二行统计卡：累计执行、成功、失败/超时；下方平均耗时
4. 若存在最近错误，红色横幅展示 `lastError`
5. 「执行历史与输出报告」双栏区：左列历史列表（状态圆点 + 触发类型 + 时间 + 耗时），右侧选中记录的输出报告
6. 左列默认选中最新一条，右侧渲染单次执行详情

**单次执行详情包含**:
- 任务上下文（任务名 + 类型图标）
- 状态头（状态图标 + 中文标签 + 触发类型徽标）
- 三宫格时间：开始 / 结束 / 耗时
- **产出结论**（摘要速览，灰底卡片）
- **报告内容**（仅当携带 Markdown 正文时，富文本渲染；如周报）
- 结构化产出（原始 JSON，调试用；有渲染报告时隐藏避免冗余）
- 错误信息（红色卡片，仅失败时）
- 元数据（traceId/token/cost 等）

**业务规则**:
- 历史按时间倒序，默认拉取最近 50 条
- 报告类任务（如周报）的完整报告正文通过 Markdown 渲染；非报告任务展示结构化 JSON

#### F-06：任务模板集

> 以下为本期提供的任务模板。模板由前端 spec 注册表与后端 handler 注册表对称维护（后续版本支持低成本新增模板，见附录 C-Q05）。

##### 模板 1：数字员工执行（agent）
- **用途**：定时触发数字员工执行一次任务并产出结论
- **配置项**：数字员工 ID（必填）、模型 ID（可选，留空用默认）、执行指令（必填）
- **频次**：全部模式可选
- **执行载体**：见附录 C-Q01

##### 模板 2：回声自检（system:echo）
- **用途**：回显参数，用于验证调度链路是否通畅
- **配置项**：参数（任意 JSON，默认 `{}`）
- **频次**：全部模式可选

##### 模板 3：报告统计（system:weekly-report，前端标签"报告统计"）
- **用途**：按周期聚合对话/Token/留存等指标，生成分析报告并留存归档
- **配置项**：分析维度（多选，见 F-07）、分析指令 Prompt、情感分析采样上限（仅吐槽/赞许维度时显示）、异常检测阈值 σ（仅突发观察维度时显示）、排行榜 Top N（仅使用排行维度时显示）、输出模板（Markdown，支持 `{{dau}}` 等占位符，留空用默认模板）
- **频次**：仅每周/每月（统计窗口由频次推导）
- **产出**：完整报告 Markdown，在执行详情中以富文本渲染
- **维度范围**：见附录 C-Q02

##### 模板 4：调用追踪清理（system:trace-cleanup）
- **用途**：清理超过 N 天的调用追踪记录（级联清理关联数据）
- **配置项**：保留天数（默认 90，必填）
- **频次**：全部模式可选

##### 模板 5：离职员工监测与清理（system:employee-cleanup）
- **用途**：监测离职/长期未活跃的数字员工，按配置清理实例、账号、记忆等
- **配置项**：
  - 离职判定方式：长期未活跃 / 实例管理服务标记离职（组件代号 claw-manager）
  - 未活跃天数（仅"长期未活跃"时显示，默认 30）
  - 监测模式：仅检测（产出离职名单，推荐）/ 检测并清理
  - 清理范围（仅"检测并清理"时显示）：数字员工实例 / 关联账号 / 记忆库数据 / 会话记录
  - 结果通知收件人（可选标签，监测/清理结果发邮件）
- **频次**：全部模式可选

#### F-07：报告分析维度库

**功能描述**: "报告统计"模板的"分析维度"字段从统一的维度库派生（依赖已定义好并完成数据埋点的平台数据指标），可多选勾选，并联动显示对应分析参数。

**维度说明**:

| 分组 | 维度口径（仅做示例，根据平台实际可统计到的口径做显示） |
|------|------|
| 基础统计 | 总览、DAU/消息量、Token/成本、留存、每日交互趋势 |
| 用户行为 | 活跃时段、粘性分层、使用排行、环比变化 |
| 质量分析 | 突发与持续观察（±Nσ 异常检测）、吐槽分析、赞许分析、AI 表现与问题 |

**业务规则**:
- 默认勾选"基础统计"组全部 + "活跃时段"
- 勾选某质量维度时联动出现其专属参数（采样上限、σ 阈值、Top N）
- 维度库是单一数据源，新增维度只需注册一条，所有引用模板自动可见
- 各维度声明其依赖的数据源，系统据此判断可产出性；前端不做硬性拦截，以保证维度可配置

## 4. 交互流程

### 4.1 主流程：创建并运行一个定时任务

```
[起点] 进入「自动化 → 定时任务」
    ↓
[1] 点击「新建任务」→ 右侧打开编辑抽屉
    ↓
[2] 填写任务名 → 选择任务模板卡片
    ↓
[3] 按模板动态渲染配置项 → 逐项填写（含字段联动）
    ↓
[4] 选择调度频次（模板可能收窄可选项）
    ↓
[判断] Cron 模式？
    ↓ 是                         ↓ 否（间隔/每天/每周/每月）
[5a] 实时校验 cron            [5b] 直接生成表达式
    ↓                              ↓
[判断] 校验通过？                ↓
    ↓ 是     ↓ 否                ↓
[6] 预览下次触发时间   [拦截] 红色错误，禁止保存
    ↓
[7] 勾选「立即启用」→ 点击「保存」
    ↓
[8] 抽屉关闭，列表刷新，任务出现（含下次执行倒计时）
    ↓
[9] 到点自动执行 / 用户点「立即执行」手动触发
    ↓
[10] 点击任务名进入详情页 → 查看「产出结论 / 报告内容」
    ↓
[结束]
```

**流程说明**:
1. 模板选择决定后续字段集；切换模板会重置表单
2. 报告统计模板会限制频次仅每周/每月，并自动推导统计窗口
3. Cron 校验为防抖实时反馈，非法则保存按钮语义上不可用（前端拦截）
4. 保存后系统按调度配置到点执行；手动执行可立即验证

### 4.2 异常分支

| 异常场景 | 触发条件 | 处理方式 | 用户提示 |
|---------|---------|---------|---------|
| 列表加载失败 | 请求异常 | 展示重试按钮 | "加载失败"+重试 |
| 任务名为空 | 必填项未填 | 阻止保存，行内错误 | "请填写任务名称" |
| 必填字段为空 | required 字段为空 | 阻止保存 | "请填写「{字段名}」" |
| JSON 格式错误 | JSON 字段解析失败 | 行内红色提示 + 阻止保存 | "存在 JSON 格式错误" / "JSON 格式错误" |
| Cron 非法 | 格式错误或语义非法 | 红色校验结果 | "✗ {错误}" |
| 执行超时 | 单次执行 >5 分钟 | 记录 timeout 状态 | 详情页状态头显示"超时"+错误信息 |
| 执行失败 | handler 抛错 | 记录 failed 状态 | 列表"上次：失败"；详情页错误卡片 |
| 被分布式锁阻塞 | 多副本同时到期 | 当前副本跳过 | 无（静默，另一副本执行） |
| 删除任务 | 用户点击删除 | 二次确认 | "确认删除任务「{name}」？执行历史将一并删除。" |
| 后端 handler 未就绪 | 模板标记 backendPending | 可保存配置但不真正执行 | "⚠ 该作业的后端 handler 尚未实现…" |

### 4.3 状态机

**对象 1：任务启用状态**

| 对象 | 状态列表 | 状态转换触发条件 |
|------|---------|---------------|
| 任务（is_enabled） | 启用 ↔ 暂停 | 用户切换启用/暂停；新建时按「立即启用」决定 |

```
[暂停] --[启用]--> [启用]（重算 nextRunAt，恢复调度）
[启用] --[暂停]--> [暂停]（停止调度，配置与历史保留）
```

**对象 2：任务最近执行结果（lastRunStatus）**

```
(无) --[首次执行成功]--> completed
(任意) --[执行成功]--> completed
(任意) --[执行失败]--> failed
(任意) --[执行超时]--> timeout
```

**对象 3：单次执行记录（run）生命周期**

```
[创建 run] --[开始执行]--> running
running   --[成功]--> completed
running   --[失败]--> failed
running   --[超时(>5min)]--> timeout
（预留）running --[手动取消/被替代]--> cancelled
```

> `pending`/`cancelled` 为状态枚举预留值：当前创建 run 即进入 `running`（不经过 `pending`）；`cancelled` 用于未来手动取消/被新执行替代场景，本期不产生。

- `triggerType`：`scheduled`（定时触发，执行后推进下次执行时间）/ `manual`（手动触发，不改变既有节奏）

## 6. 数据模型

> 本章描述产品实体概念与关键字段含义（非 DB 实现细节）。

### 6.1 核心实体

```typescript
// 定时任务（配置实体）
interface ScheduledTask {
  id: string;                  // 任务唯一标识
  name: string;                // 任务名称（≤128 字符）
  description: string | null;  // 描述（可选）
  jobType: 'agent' | 'system'; // 任务类型
  jobPayload: Record<string, unknown>; // 自描述配置（含 handlerKey 等）
  scheduleType: 'cron' | 'interval';   // 调度方式
  cronExpr: string | null;     // cron 表达式（scheduleType=cron 时）
  intervalSeconds: number | null; // 间隔秒数（scheduleType=interval 时）
  timezone: string;            // 时区（默认 Asia/Shanghai）
  isEnabled: boolean;          // 是否启用
  nextRunAt: string | null;    // 下次到期时间（调度器判定依据）
  lastRunAt: string | null;    // 上次执行时间
  lastRunStatus: 'completed' | 'failed' | 'timeout' | null; // 上次结果
  lastError: string | null;    // 最近错误信息
  createdBy: string | null;    // 创建人
  createdAt: string;
  updatedAt: string;
}

// 单次执行记录（观测实体）
interface ScheduledTaskRun {
  id: string;
  taskId: string;              // 所属任务（级联删除）
  status: 'pending' | 'running' | 'completed' | 'failed' | 'timeout' | 'cancelled';
  triggerType: 'scheduled' | 'manual'; // 触发类型
  startedAt: string | null;    // 开始时间
  finishedAt: string | null;   // 结束时间
  durationMs: number | null;   // 耗时
  conclusion: string | null;   // 产出结论（展示主体）
  outputPayload: unknown;      // 结构化产出（报告正文 Markdown 等放此）
  errorMessage: string | null; // 错误信息（失败/超时）
  metadata: unknown;           // 扩展元数据（traceId/token/cost）
  createdAt: string;
}

// 任务模板（前端注册表，驱动配置表单渲染）
interface JobSpec {
  id: string;                  // 'agent' | 'system:<handlerKey>'
  jobType: 'agent' | 'system';
  handlerKey?: string;         // system 任务的处理器标识
  label: string;               // 模板卡片名称
  icon: string;                // 图标
  description: string;         // 一句话描述
  fields: JobFieldSpec[];      // 配置字段（含类型/默认值/必填/联动）
  allowedFreqModes?: FreqMode[]; // 限制可选频次
  backendPending?: boolean;    // 标记执行能力未就绪
}

// 报告维度（报告统计模板的分析维度库）
interface ReportDimension {
  id: string;
  label: string;
  group: 'basic' | 'engagement' | 'quality';
  requires?: string[];         // 依赖的数据源
  description: string;
}
```

### 6.2 字段要点说明

| 字段 | 类型 | 必填 | 说明 | 约束 |
|------|------|------|------|------|
| name | string | 是 | 任务名称 | ≤128 字符 |
| jobType | enum | 是 | agent / system | 二选一 |
| scheduleType | enum | 是 | cron / interval | 二选一 |
| cronExpr | string | 否 | cron 表达式 | scheduleType=cron 时必填，5 段式 |
| intervalSeconds | number | 否 | 间隔秒数 | scheduleType=interval 时必填，≥1 |
| nextRunAt | datetime | 否 | 下次到期 | 为空表示无调度（暂停或无效配置） |
| status（run） | enum | 是 | 执行状态 | pending/running/completed/failed/timeout/cancelled |
| triggerType | enum | 是 | 触发类型 | scheduled 推进节奏，manual 不推进 |
| conclusion | text | 否 | 产出结论 | 列表/详情展示主体 |

> **约束**：单次执行默认超时上限 5 分钟，超时记 `timeout`。

### 6.3 实体关系

```
[ScheduledTask] 1 --- N [ScheduledTaskRun]
        ↑
        | 引用模板渲染配置（多对一，逻辑引用）
        |
   [JobSpec]（前端注册表）
        |
   [ReportDimension]（报告统计模板引用维度库）
```

**关系说明**:
| 关系 | 类型 | 说明 |
|------|------|------|
| ScheduledTask → ScheduledTaskRun | 一对多 | 一个任务的每次执行为一条 run；任务删除级联删除其历史 |
| ScheduledTask → JobSpec | 多对一（逻辑） | 任务通过 jobType + handlerKey 反查模板，用于渲染配置与详情 |
| JobSpec（报告统计）→ ReportDimension | 多对多（引用） | 报告模板的分析维度字段从维度库派生 |

|-----|--------|------|
| 调度可靠性 | 停机期间到期任务重启后补执行 | 启动即跑一次 tick |
| 并发安全 | 同任务同时刻不重复执行 | PG advisory lock + 单进程 in-flight 互斥 |
| 故障隔离 | 单任务失败不影响其他任务与调度 | try/catch 隔离每个任务 |

### 8.3 安全

- 所有管理操作通过鉴权中间件保护
- 删除等危险操作二次确认
- 分布式锁防止重复执行
- 敏感配置（邮件凭证等）加密存储，未配置时静默不报错

### 8.4 可扩展性

| 维度 | 目标 | 实现方式 |
|------|------|---------|
| 新增作业类型 | 后续版本支持低成本扩展 | 前端 spec 注册表 + 后端 handler 注册表对称扩展（见附录 C-Q05） |
| 报告维度扩展 | 注册一条即全局可见 | 维度库单一数据源 |
| 多副本 | 水平扩展安全 | DB-tick + advisory lock |
| 执行器替换 | 任务执行载体可平滑替换 | 接口隔离，仅替换实现 |

### 8.5 可维护性

| 维度 | 要求 |
|------|------|
| 模块化 | 调度器 / 锁 / handler 注册表 / 模板注册表 各司其职 |
| 可观测 | 每次执行留存 conclusion + 结构化产出 + 元数据（traceId/token/cost） |
| 配置先行 | 配置能力与执行能力解耦，配置可先于执行能力就绪 |

### 附录 A：术语表

| 术语 | 定义 |
|-----|------|
| 定时任务（ScheduledTask） | 一条周期性自动执行的配置，含做什么（payload）与多久做一次（schedule） |

| 系统作业（system） | 平台自身周期动作（清理/监测/报告/自检） |
| DB-tick | 数据库驱动的统一调度心跳，周期性扫描到期任务 |
| advisory lock | PostgreSQL 咨询锁，保证多副本下同任务同时刻只执行一次 |
| 产出结论（conclusion） | 一次执行的核心文字结论，为列表/详情展示主体 |
| in-flight 互斥 | 单进程内同一任务执行期间不被下一 tick 重入 |
