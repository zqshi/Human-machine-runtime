## 1. 功能概述

### 1.1 功能定位

当前记忆库存在三个核心问题：

1. **部门隔离缺失**：所有数字员工的记忆库混在同一空间，无法按部门筛选和管理
2. **共享记忆包含画像类型**：画像是个体偏好，不应泛化为共享知识；共享记忆应仅包含通用规则
3. **生命周期脱节**：记忆库是创建数字员工时的附属产物，但记忆库管理页可独立删除，导致员工详情页状态不一致

**核心业务事实**：一个部门就一个 Agent，服务于多个用户。用户产生个人记忆，Agent 对个人记忆做共性抽象+脱敏，形成该 Agent 的共享记忆（通用规则）。

本次升级：记忆库按部门归属隔离、共享记忆限定为规则类型、新增共识提取规则类型、记忆库生命周期收归员工详情页控制。

### 1.2 目标用户

| 用户类型 | 特征描述 | 核心诉求 |
|---------|---------|---------|
| 管理员 | 管理后台操作者，负责配置数字员工和记忆库 | 按部门管理记忆；记忆库启停与员工状态一致 |

### 1.3 核心价值

| 维度 | 描述 |
|------|------|
| 用户价值 | 记忆库按部门归属，管理员可快速定位特定部门的 Agent 记忆配置；共享记忆仅包含通用规则，不会将个体偏好错误推给其他用户 |
| 业务价值 | 部门归属让记忆按业务线隔离；共享记忆限定为规则类型，提升 Agent 回答一致性 |
| 技术价值 | 对齐 Mem0 多租户隔离模型（org/project/agent/user）；记忆库启停与数字员工状态双向联动（Mem0 映射详见 6.3） |

### 2.3 详细规格

#### F-01：记忆库列表部门筛选

**功能描述**: 在记忆库列表页顶部增加部门筛选器，管理员可按部门过滤记忆库。部门列表从数字员工归属的部门实体生成。

**前置条件**:
- 至少有一个数字员工归属了部门

**主要行为**:
1. 页面统计卡片下方显示部门筛选下拉框
2. 下拉框列出所有部门（去重），支持搜索
3. 选择部门后，列表仅显示该部门下的记忆库
4. 清空筛选恢复全量展示

**业务规则**:
- 部门筛选为可选项，默认展示全部
- 未归属部门的数字员工，其记忆库在选择"全部"时展示，归类为"未分配"
- 统计卡片随筛选结果动态更新

**边界条件**:

| 边界场景 | 处理方式 |
|---------|---------|
| 无任何部门数据 | 筛选器隐藏，列表正常展示 |
| 记忆库对应员工未归属部门 | 归类为"未分配"，可被"全部"筛出 |
| 部门下无记忆库 | 展示空状态提示 |

---

#### F-02：记忆库部门自动归属

**功能描述**: 记忆库不独立创建，而是在创建数字员工时勾选"启用记忆库"自动创建。创建后，记忆库自动继承该数字员工的部门归属。

**前置条件**:
- 无（与现有创建流程一致）

**主要行为**:
1. 管理员在创建数字员工向导的"基础信息"步骤勾选"启用记忆库"
2. 通过部门选择器选择归属部门（下拉当前已有部门 + 内联「新建部门」）
3. 提交创建后，系统自动创建记忆库，其部门归属 = 该数字员工的部门
4. 记忆库列表展示部门标签

**业务规则**:
- 记忆库的部门归属由数字员工的部门决定，不可独立修改
- 若数字员工的部门变更，其记忆库的部门归属随之变更
- 未归属部门的数字员工勾选启用记忆库时，记忆库部门为"未分配"

---

#### F-03：员工详情页记忆库状态展示

**功能描述**: 数字员工详情页增加记忆库状态区域，展示该员工是否有记忆库、启用状态、记忆片段数量。

**前置条件**:
- 无

**主要行为**:
1. 员工详情页"基础信息"区域增加"记忆库"状态行
2. 若有记忆库(已启用)：显示"已启用"+ 个人记忆片段数 + 共享片段数
3. 若有记忆库(已归档)：显示"已禁用"+ 记忆片段数（归档）
4. 若无记忆库：显示"未启用"+ "启用"按钮
5. "记忆库"按钮保留，点击跳转记忆库详情页（若有库）或提示启用

**业务规则**:
- 状态实时查询，不缓存
- 无记忆库时，"记忆库"按钮点击提示"该员工尚未启用记忆库，是否现在启用？"

---

#### F-04：记忆库页面移除删除按钮

**功能描述**: 记忆库是数字员工的附属产物，生命周期应由数字员工侧控制。记忆库管理页面的职责是管理记忆内容（片段、规则、检索验证），不控制记忆库的创建和删除。

**主要行为**:
1. 记忆库列表页每行的操作按钮从"查看 | 删除"改为"查看"
2. 记忆库详情页不提供删除或归档入口
3. 如需删除或关闭记忆库，引导至对应数字员工详情页操作

**业务规则**:
- 记忆库管理页 = 内容管理（查看 + 增删片段/规则）
- 员工详情页 = 生命周期管理（启用/关闭/删除）

---

#### F-05：员工详情页关闭记忆库

**功能描述**: 员工详情页可关闭记忆库。关闭时，系统判断该记忆库是否有记忆数据，根据情况引导用户选择保留或删除。

**前置条件**:
- 数字员工已有记忆库（状态为已启用）

**主要行为**:
1. 员工详情页记忆库状态区域展示"关闭记忆库"按钮
2. 点击关闭后，系统检查记忆库中是否有记忆数据：
   - **无数据**（空库）：弹窗确认"关闭记忆库？当前无记忆数据，关闭后将不再为用户存储记忆。"确认后直接删除空库
   - **有数据**：弹窗展示片段数和用户覆盖数，让用户选择：
     - **保留记忆库（归档）**：记忆数据保留，Agent 不再检索记忆，后续可重新启用
     - **删除记忆库**：所有记忆数据永久删除，不可恢复，后续重新启用时创建空库
3. 选择后执行对应操作，员工详情页状态更新

**启用行为**:
1. 点击"启用记忆库"
2. 若存在归档库：
   - **有数据**：弹窗提示"将恢复 X 条历史记忆（个人 Y 条 / 共享 Z 条）"；若归档期间员工部门已变更，额外提示"该员工部门已变更，历史记忆仍为原部门期间产生"；用户确认后恢复为已启用
   - **无数据**（空归档库）：直接恢复为已启用
3. 若不存在归档库：创建空记忆库，部门归属继承员工部门
4. 状态变为"已启用"

**业务规则**:
- 关闭 ≠ 删除。保留=归档（可逆），删除=不可逆
- 有数据时关闭必须让用户选择，不可静默归档或静默删除
- **归档期间库冻结**：归档记忆库既不检索（读停）也不写入新记忆（写停），归档 = 记忆快照；恢复后回到归档时刻的状态
- **恢复需确认**：归档库有数据时，恢复前弹窗告知将恢复的记忆数量与部门变更情况，不可静默恢复
- 删除后重新启用，创建空库，之前的数据不可恢复
- 操作记录审计日志

---

#### F-06：共享记忆限定为规则类型

**功能描述**: 共享记忆是 Agent 从各用户个人记忆中提取的脱敏通用知识，供所有用户共享。画像(preference)本质是个体偏好，不应泛化为共享知识；只有规则(fact)才是通用事实，人人适用。因此共享记忆的类型锁定为"规则"，移除"画像"选项。

**改动点**:
1. **共享记忆 Tab 手动补充**：表单中移除类型选择，类型固定为"规则"
2. **个人记忆 Tab 添加**：保留"画像/规则"双选项（个人记忆可以含画像）
3. **共识提取规则**：提取结果仅包含规则类型，忽略画像
4. **检索验证**：共享记忆命中结果类型标签固定显示"规则"

**业务规则**:
- 个人记忆：画像 + 规则（因人而异，含偏好和事实）
- 共享记忆：仅规则（通用事实，人人适用）
- 共识提取中明确：忽略画像类型，仅提取规则类共识

---

#### F-07a：共识提取规则配置

**功能描述**: 在"记忆规则"Tab 中新增"共识提取"规则类型，与现有"片段规则"、"画像规则"并列。本功能交付规则配置界面，执行引擎见 F-07b。

**页面变化**:
1. 规则类型下拉框新增"共识提取"选项
2. 选择"共识提取"后，规则配置表单展示专属字段：
   - 触发条件：对话结束 / 收到反馈 / 定时
   - 最低用户数：≥ N 个用户出现相同记忆才提取（默认 2）
3. 规则列表中"共识提取"类型规则展示专属标签（绿色"共识"标签）
4. 启用/禁用/删除操作与现有规则一致

**业务规则**:
- F-07a 仅持久化规则配置，不执行提取
- 提取执行依赖 F-07b。规则启用后暂无实际提取行为，需在 UI 提示"执行引擎待上线"

---

#### F-07b：共识提取执行引擎

**功能描述**: 当 F-07a 配置的"共识提取"规则被触发时，系统扫描该 Agent 下所有用户的个人记忆，识别 ≥ 最低用户数的规则类共识，脱敏后写入共享记忆。

**执行流程**:
1. 触发条件命中（对话结束 / 收到反馈 / 定时）
2. 扫描该 Agent 下所有用户的个人记忆
3. LLM 共识识别：仅提取规则类型共识，忽略画像；合并相似表达；标注用户数/置信度
4. 脱敏处理：移除姓名/账号/联系方式等个人标识
5. 直接写入共享记忆

**业务规则**:
- 仅提取规则类共识，忽略画像
- 提取结果必须脱敏
- 直接写入共享记忆（Agent 级通用规则）

---

#### F-08：部门实体化

**功能描述**: 部门从自由文本升级为独立实体，数字员工归属部门。本次升级让部门成为可管理的一等对象（可新建、选择、跨员工复用），为按部门筛选/管理记忆库提供基础。

**主要行为**:
1. 部门是租户内独立实体，支持增删改查；租户内部门名称唯一
2. 创建数字员工时，"归属部门"由自由文本输入改为**部门选择器**：下拉选择已有部门 + 内联「新建部门」
3. 数字员工通过部门归属关联，一个部门可含多个数字员工
4. 部门归属驱动记忆库按部门筛选/展示（F-01）

**业务规则**:
- 部门是租户内实体，跨租户允许同名
- 删除部门时，归属该部门的数字员工部门置空，不级联删除员工
- 部门选择器输出部门实体；为兼容历史，部门名称文本同步保留

---

#### F-09：部门级跨 Agent 共享记忆（未来规划，本期不含）

**功能描述**: 部门级跨 Agent 共享记忆——同部门所有数字员工（Agent）通用的脱敏规则，检索时与个人记忆、Agent 共享三层合并。

**本期说明**: 本期产品形态为「一个部门一个 Agent」，无跨 Agent 共享场景，**本期不含此功能**。作为未来规划保留设计方向，待后续出现"一个部门多个 Agent"需求时开放。

**未来目标态**:
1. 记忆库详情页增加「部门共享」Tab
2. Tab 内展示/补充部门共享规则（仅规则类型）
3. 检索时三层合并（个人 / Agent 共享 / 部门共享），部门共享权重最低
4. 涉及的 Mem0 维度（app_id）详见 6.3

## 4. 交互流程

### 4.1 主流程 — 员工详情页关闭记忆库

```
[起始点] 管理员在员工详情页点击"关闭记忆库"
    ↓
[判断] 记忆库中是否有记忆数据？
    ↓ [无数据]                              ↓ [有数据]
[步骤1a] 弹窗确认                        [步骤1b] 弹窗展示片段数+用户覆盖数，让用户选择：
  "关闭记忆库？当前无记忆数据，               ├─ 保留记忆库（归档）
   关闭后将不再为用户存储记忆。"              └─ 删除记忆库（不可恢复）
    ↓                                     ↓
[确认] 直接删除空库                    [选择保留]               [选择删除]
    ↓                                     ↓                        ↓
状态→未启用                           状态→已归档              删除库+数据
                                          ↓                        ↓
                                    Agent 不再检索记忆        状态→未启用
                                    数据保留，可重新启用      后续启用创建空库
```

### 4.2 主流程 — 员工详情页启用记忆库

```
[起始点] 管理员在员工详情页点击"启用记忆库"
    ↓
[判断] 是否存在归档的记忆库？
    ↓ [存在归档]                              ↓ [不存在]
[判断] 归档库是否有数据？                    [步骤] 创建空记忆库
    ↓ [有数据]              ↓ [无数据]              ↓
[弹窗] 提示将恢复          直接恢复           部门归属继承员工部门
  X 条历史记忆           归档库为已启用             ↓
  + 部门变更提示(若有)        ↓                状态→已启用
    ↓ [确认]              状态→已启用
[恢复] 归档库为已启用
  记忆数据恢复可检索
    ↓
状态→已启用
```

**流程说明**:
1. 归档库有数据时必须用户确认，弹窗告知将恢复的记忆数量与部门变更情况（对应 F-05 恢复需确认规则）
2. 归档期间库冻结（不读不写），恢复即解冻，回到归档时刻状态

### 4.3 主流程 — 共识提取规则配置与执行

> 配置部分（步骤1~3）= F-07a；执行部分（触发~写入）= F-07b。

```
[起始点] 管理员在记忆规则 Tab 创建"共识提取"规则
    ↓
[步骤1] 选择规则类型"共识提取"
    ↓
[步骤2] 配置触发条件、最低用户数
    ↓
[步骤3] 启用规则
    ↓ ══════════════════════════════════════════════
[触发] 对话结束 / 收到反馈 / 定时
    ↓
[步骤4] 系统扫描该 Agent 下所有用户个人记忆
    ↓
[步骤5] 识别 ≥ 最低用户数的共识模式（仅规则类型，忽略画像）
    ↓
[步骤6] 脱敏处理
    ↓
[步骤7] 直接写入共享记忆（Agent 共享）
```

### 4.4 异常分支

| 异常场景 | 触发条件 | 处理方式 | 用户提示 |
|---------|---------|---------|---------|
| 记忆库已被删除 | 员工详情页点击"记忆库"但库不存在 | 展示"未启用"状态，提供启用按钮 | "该员工的记忆库已被删除" |
| 关闭记忆库后 Agent 对话中 | Agent 正在对话时记忆库被归档/删除 | 当前对话继续，下次对话不再检索记忆 | 无（无感知降级） |
| 部门下无记忆库 | 筛选某部门但无记忆库 | 展示空状态 | "该部门暂无启用记忆库的数字员工" |
| 记忆库创建失败 | 启用时创建失败 | 回滚状态为"未启用"，提示重试 | "记忆库创建失败，请重试" |

### 4.5 状态机 — 记忆库生命周期

```
[不存在] ──启用──→ [已启用] ──关闭(选择保留)──→ [已归档]
                       ↑                              │
                       └──────启用(恢复)──────────────┘

[已启用]   ──关闭(选择删除)──→ [不存在]（不可逆）
[已启用]   ──关闭(无数据)──→ [不存在]（空库直接删除）
```

| 状态 | 记忆数据 | Agent 是否检索 | 员工详情页显示 | 操作 |
|------|---------|:---:|---------|------|
| 不存在 | 无 | 否 | "未启用"+ 启用按钮 | 启用→创建空库 |
| 已启用 | 有 | 是 | "已启用"+ 关闭按钮 | 关闭→选择保留/删除 |
| 已归档 | 有（保留） | 否 | "已禁用"+ 启用按钮 | 启用→恢复已启用 |

## 6. 数据模型

### 6.1 核心实体

| 实体 | 说明 | 关键属性 |
|------|------|---------|
| 部门 | 租户内独立实体，记忆按部门归属/筛选 | 名称（租户内唯一）、描述 |
| 数字员工 | 记忆库的拥有者 | 名称、归属部门、是否启用记忆 |
| 记忆库 | Agent 的记忆容器，与数字员工 1:0..1 | 所属数字员工、状态（已启用/已归档）、检索配置 |
| 个人记忆 | 用户级记忆片段 | 所属记忆库、用户ID、类型（画像/规则）、内容、来源 |
| 共享记忆（Agent 级） | Agent 的脱敏通用规则，仅规则类型 | 所属记忆库 + Agent；人人适用 |
| 记忆规则 | 记忆自动处理规则 | 所属记忆库、规则类型（片段/画像/共识提取）、触发条件 |

> 部门是记忆库的归属维度（筛选/展示）；记忆库本身不单独存储部门，部门归属来自所属数字员工。

### 6.2 记忆层级与类型约束

本期形态为「一个部门一个 Agent」，记忆层级为：

```
部门（归属维度，用于筛选/管理）
  └─ Agent（一个数字员工）
       ├─ Agent 共享记忆（仅规则，人人适用）
       ├─ 用户 A 个人记忆（画像 + 规则）
       └─ 用户 B 个人记忆（画像 + 规则）
```

| 记忆层级 | 允许的类型 | 理由 |
|---------|-----------|------|
| 个人记忆 | 画像 + 规则 | 因人而异，含个人偏好和事实 |
| Agent 共享 | 仅规则 | 一个 Agent 内多用户共性 |

> 跨 Agent 的"部门共享记忆"层（F-09）本期不含，详见 F-09 与 6.3。

### 6.3 Mem0 字段映射

> 本节为研发理解记忆隔离模型而保留。Mem0 采用 `Organization → Project → Entity` 三级隔离，Memory 归属到具体 Entity。

**Mem0 官方层级结构**:

```
Organization (org_id)                      ← SaaS 租户隔离
  └─ Project (project_id)                  ← 业务隔离维度（本期 = 部门）
       ├─ Entity: User   (user_id)         ← 终端用户
       ├─ Entity: Agent  (agent_id)        ← 数字员工
       └─ Entity: App    (app_id)          ← 应用级共享实体
            └─ Memories (+ 可选 run_id 会话级)
```

> `user_id` / `agent_id` / `app_id` 同属 Project 下的 Entity 层，三者并列；写入记忆时"传哪个 Entity"决定记忆归属范围（传 user_id = 个人，传 agent_id = Agent 共享，传 app_id = 部门共享）。

**产品概念 → Mem0 字段映射**:

| 产品概念 | Mem0 层级 | Mem0 字段 | 值来源 | 本期启用 | 说明 |
|---------|-----------|-----------|--------|:--------:|------|
| 租户/组织 | Organization | `org_id` | 租户 ID | ✅ | SaaS 客户隔离 |
| 部门 | Project | `project_id` | 部门 ID | ✅ | 部门归属维度（本期"一部门一 Agent"下与 agent_id 隔离冗余，保留为部门维度） |
| 数字员工 | Entity (Agent) | `agent_id` | 数字员工 ID | ✅ | Agent 级隔离 + Agent 共享记忆归属 |
| 终端用户 | Entity (User) | `user_id` | 终端用户 ID | ✅ | 个人记忆隔离 |
| 部门共享 | Entity (App) | `app_id` | `app_<部门ID>` | ⏸️ | 跨 Agent 共享（F-09，本期非需求，未启用） |
| 会话 | — | `run_id` | 会话 ID | ❌ | 长/短期记忆隔离，待排期 |

**各记忆类型的 Mem0 调用方式**:

| 记忆类型 | `add()` 传参 | `search()` 传参 | 本期 |
|---------|-------------|----------------|:----:|
| 个人记忆 | `user_id` + `agent_id` + `org_id`（+ `project_id`） | 同左 | ✅ |
| Agent 共享 | `agent_id` + `org_id`（不传 `user_id`） | 同左 | ✅ |
| 部门共享 | `app_id` + `org_id`（不传 `agent_id`/`user_id`） | 同左 | ⏸️ |

### 6.4 记忆库生命周期

```
数字员工 ──── 1:0..1 ────→ 记忆库
                                │
                          status: 已启用 | 已归档
                                │
                    1:N ──────→ 记忆片段
                    1:N ──────→ 记忆规则
```

### 6.5 检索优先级

用户对话时，Agent 按**多层并行检索 + 加权合并**取 TopK：

| 层级 | 权重 | 说明 |
|------|------|------|
| 个人记忆 | ×1.0 | 该用户专属记忆，权重最大 |
| Agent 共享 | ×0.85 | 该 Agent 的脱敏通用规则 |
| 部门共享 | ×0.7 | 部门级跨 Agent 共享（F-09，本期未启用） |

> 合并规则：多层并行检索，按 `得分 × 权重` 合并；同一命中跨层取最高分，附加来源标记；按合并分排序取 TopK。本期实际为个人 + Agent 两层。

| 记忆库归档 | 数据保留，可恢复 | 关闭≠删除 |

### 8.3 安全

- 所有 API 接口通过鉴权中间件保护
- 租户隔离：按租户强制隔离，跨租户不可访问
- 部门隔离：本期「一部门一 Agent」，按 Agent 天然隔离
- 共识提取脱敏：LLM 输出经脱敏处理，不含个人标识
- 操作审计：记忆库启停/关闭/删除有审计日志

### 8.4 可扩展性

| 单用户记忆条目 | 支持 10000+ | 分页 + 索引 |
| 多租户 | 支持 | 按租户隔离 |

### 附录E：共享记忆 Prompt 清单（F-07b 设计稿）

> 以下 Prompt 为共识提取执行引擎的设计稿，供 F-07b 实现参考。

| Prompt 用途 | 使用场景 | 输入 | 输出 |
|------------|---------|------|------|
| 共识识别 | 从多用户个人记忆中提取共同模式 | 一组用户记忆片段 | 脱敏共识列表（仅规则类型，含用户数、置信度、证据片段） |
| 隐私脱敏 | 对提取结果进行隐私保护处理 | 一条待脱敏的记忆内容 | 脱敏后内容 + 是否可共享标记 |

**共识识别 Prompt**:

```
你是一个记忆分析专家。以下是同一 Agent 内多个用户的个人记忆片段。请从中识别出多个用户共同出现的共识性知识。

输入：一组用户记忆片段，每条包含用户ID、记忆类型、记忆内容

规则：
1. 仅提取规则类型(fact)的共识，忽略画像类型(preference)——画像是个体偏好，不可泛化为共享知识
2. 仅提取至少N个不同用户共同提及的共识，忽略仅单用户独有的记忆（N为规则配置的最低用户数）
3. 提取结果必须脱敏：移除所有用户姓名、账号、个人标识符
4. 将相似表达合并为一条简洁的共识陈述
5. 对每条共识标注：涉及的用户数量、置信度（高/中/低）

输出格式（JSON）：
{
  "consensus_memories": [
    {
      "content": "脱敏后的共识内容",
      "type": "fact",
      "source_user_count": 3,
      "confidence": "high | medium | low",
      "evidence_snippets": ["相关原文片段1（脱敏）", "相关原文片段2（脱敏）"]
    }
  ]
}
```

**脱敏 Prompt**:

```
你是一个数据脱敏专家。以下是一条从用户个人记忆中提取的共识性知识。请确保其中不包含任何个人隐私信息。

输入：一条待脱敏的共享记忆内容

脱敏规则：
1. 将所有用户姓名替换为"用户"
2. 将所有具体账号/ID替换为"[已脱敏]"
3. 将所有个人联系方式（手机/邮箱/地址）替换为"[已脱敏]"
4. 保留业务事实和规则信息不变
5. 如果脱敏后信息量严重丢失，标记该条为"不适合共享"

输出格式（JSON）：
{
  "sanitized_content": "脱敏后的内容",
  "is_shareable": true | false,
  "removed_items": ["移除了1个手机号", "移除了1个邮箱"]
}
```
