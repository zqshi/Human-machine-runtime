## 版本目标

当前架构经实证是"声明式 + 运行时动态组装"，缺编译固化层（`grep bake/compile/固化` 全空），运行时每次 dispatch 动态查 DB 拼装 prompt/tools/skills（harness.ts:179-244），存在运行时漂移 + 性能开销 + 线上 Agent 行为不可锁定问题。本版本：

1. 新增编译固化层：AgentDefinition 发布时 bake 成不可变 RuntimeManifest，运行时读 manifest 不再动态查 DB。
2. CubeSandbox（腾讯云 KVM MicroVM）作为高安全可选执行层与 OpenSandbox 并存。
3. 管理后台获编译固化管理 + 沙箱策略配置 + Agent 向导固化流程能力。

## 版本验证标准

1. Agent 声明发布 → bake 生成 RuntimeManifest（status=baked，不可变）；运行时 harness 读 manifest 拿 systemPrompt/tools/skills，不再调 assemble+getPersona。
2. 未 bake 的存量 Agent 走降级老路径（assemble+getPersona），行为不变（灰度兼容）。
3. sandboxTemplate='kvm-microvm' 的 Agent，其 sandbox 工具调用路由到 CubeSandboxExecutor（KVM 隔离），而非 OpenSandboxExecutor。
4. 管理后台能触发 bake、查看 manifest 内容、generation 版本对比、回滚；能配置 CubeSandbox 网关与沙箱选择策略。
5. 修改 spec → bumpGeneration → re-bake 新 manifest，旧 instance 引用旧 generation manifest（灰度/回滚）。

## 版本范围

**做什么:**
- Layer 2 编译固化层：RuntimeManifest domain + agent_runtime_manifests 表 + BakingService + IRuntimeManifestPort

- 不废弃 assembly-provider/personaProvider（保留灰度降级，bake 普及后单独收敛）→ 后续版本

## 各任务实施要点

### C1 — RuntimeManifest domain
固化字段见设计文档 §3.1。`sealManifest()` 递归 `Object.freeze`。零外部依赖（domain 纪律）。含编译态校验（compiledTools 非空校验、quota 快照完整性）。单测 100%（§2.2 domain 必测）。

### C2 — DB 表 + migration
唯一约束 `(agentDefinitionId, generation)`；索引 `agentDefinitionId` + `status`。migration 必须 `.ts`（§7.2.1 第3条，migrate.ts 不跑 .sql，tsc/vitest 测不出缺列）。repository 仿 `sandbox-session-repository.ts`（findSandboxSession 模式）。

### C3 — BakingService
复用 C4 提纯的 assembleTools/assembleSkills（不重发明，§9.7）。同步固化（返回终态 baked|failed），规模大时再迁 scheduler 异步队列。任一步失败 → status=failed + errorMsg，不污染 baked manifest。注入 IAgentDefinitionPort（已有，assembly-provider.ts:18）。

### C4 — assembleTools/assembleSkills 提纯
当前是 AssemblyProvider 实例方法（assembly-provider.ts:187/260）。提为可复用纯函数或独立 module，bake（C3）与运行时（降级路径 C6）共用。行为不变（跨租户校验/禁用跳过/inputSchema 缺省降级/空数组语义陷阱），有 assembly-provider.test.ts 守护。

### C5 — IRuntimeManifestPort
仿 IInstanceLookupPort（assembly-provider.ts:13）模式，消费方定义 port 守 §1.3（T47 同款）。`getManifest(instanceId): Promise<RuntimeManifest|null>`（instance→agentDefinitionId+generation→查）。bootstrap 适配注入（仿 adaptInstanceLookup）。

### C6 — harness 改造（灰度关键）
harness.ts:179-244：优先读 manifest（status=baked 才用），无则降级 assemble+getPersona。**不可跳过降级分支**（否则存量未 bake Agent 全挂）。运行时仍动态组装：凭证/session/实时配额/用户输入/trace（不从 manifest 读）。

### C7 — sandboxTemplate 扩展
agent-definition.ts:113 枚举加 'kvm-microvm'，validateAgentDefinitionSpec（:117）同步校验。

### C8 — CubeSandboxExecutor
平行 OpenSandboxExecutor（opensandbox-executor.ts:85）。IToolExecutor.execute 按 op 分流。E2B Node SDK（`e2b` 包），端点指向 CubeSandbox CubeAPI 网关。复用 getOrCreateSandbox 三段式 + mapSearchEntries + resolveWorkspacePath。

### C9 — SandboxRouter + executor-factory 改造
当前 executor-factory.ts:23 sandbox 单例二选一，改为 SandboxRouter 按 sandboxStrategy 路由（kvm-microvm→CubeSandbox，其余→OpenSandbox）。sandboxStrategy 来源：params.__callerId→instance→manifest，或 tool config.sandboxBackend（bake 固化）。

### C10 — sandbox_sessions backend 字段
复用 T55 的 sandbox_sessions 表（callerId↔sandboxId），加 backend（opensandbox/cubesandbox）区分。migration .ts。

### C11 — config + bootstrap 装配
config 加 cubesandbox（domain/apiKey/image）+ .env.example（CUBESANDBOX_DOMAIN/CUBESANDBOX_API_KEY）。bootstrap 装配 CubeSandboxExecutor + SandboxRouter + BakingService + RuntimeManifestPort 适配。AppContext（types.ts）加字段。

### C12-C14 — 后台
复用 T46/T47 模式：有编排逻辑（bake 触发/版本对比/回滚）建 use-case+test；逻辑薄（manifest 查看/沙箱配置 CRUD）用 adminApi re-export。前端 Section 挂 AdminShell（仿 T44 CredentialSection）。路由挂 authMiddleware（§7.2.1 第1条）。

### C15 — 文档 + 门禁
设计文档 v2.0-declarative-baking-runtime.md 每节 [PLANNED]→[IMPLEMENTED]。§13 会话/任务级巡检 + §14 版本归档门禁（12 必修项）。
