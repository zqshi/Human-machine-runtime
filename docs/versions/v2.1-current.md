## 版本目标

cockpit 当前是"零 domain"状态：无独立 context（全部堆 `routes/cockpit/` 17 文件）+ `cockpitEntities` EAV 通用单表（`id/entityType/tenantId/data(jsonb)`，五子系统靠 entityType 字符串区分、领域语义塞无类型 jsonb，无实体不变式、无值对象）+ `CockpitRepository` 弱类型 CRUD 挂 `db/repositories/` 非 context。本版本：

1. 建 `server/src/contexts/cockpit/`（domain/application/adapters），把 routes/cockpit 业务逻辑下沉。
2. 五子系统实体从 EAV 抽出为带不变式+值对象的 domain 实体（Objective/Signal/Decision/Orchestration/Evaluation），对标前端 `client-suite domain/` 已有实体（避免前后端 domain 分裂，守 §6.4）。
3. dispatch trace→涌现信号数据回流（自动提取，替代手动录入；感知是神经系统，没它上面三层全断裂）。
4. 判断节点识别接 LLM + 编排接 /agent/dispatch 真路由 + correction 传播生效。

## 版本验证标准

1. cockpit 有独立 context（domain/application/adapters 分层），routes/cockpit 为薄层（参数校验+转发，无业务判断，守 §12信号6）。
2. 五子系统实体带不变式（如 Signal.urgency 限定枚举、Objective.confidence ∈ [0,1]）+ 值对象，不再是 jsonb blob。
3. 涌现信号从 dispatch trace 自动提取（非手动录入 / 非 LLM 编造），/corrections/apply effective 反映真实传播。
4. 前端 DTO 不变（DecodedStrategyDTO 等结构保持，cockpitObjectiveApiAdapter 无需改），前后端 domain 对齐。

### E10（done）— 判断子系统 domain 建模

对标前端**实现态 DTO 契约**（`cockpitApiAdapter.toDecisionRequest` / `JudgmentRecord.rehydrate`），非设计态超前实体（不建 DTO 没有的 `sourceNotificationId`/`relatedTaskIds`，守 Phase B 教训 §6.3）。

- **schema**：decisions 表（recommendation/alternatives jsonb 值对象、urgency/responseStatus 枚举列、deadline/responseAt bigint ms、impactScope/downstreamTaskIds/downstreamGoalIds、tenantId、createdAt/updatedAt timestamptz + 4 index）+ judgment_records 表（decisionId/source/action 枚举列、respondedAt/createdAt bigint ms、contextSnapshot jsonb 快照 + 3 index）。
- **migration**：CREATE/INDEX/INSERT 全 IF NOT EXISTS/ON CONFLICT 幂等；EAV `entity_type IN ('decision','judgment_record')` 迁移，camelCase→snake_case。实测无源数据迁 0 行无错。
- **domain**：Decision（immutable DDD，respond 状态机返回新实例，urgency/responseStatus 枚举校验，impactScope clamp >=0）+ JudgmentRecord（rehydrate 容错脏枚举 + fromDecisionResponse 从已响应 decision 生成审计 record）+ JudgmentAnalytics（1:1 复刻前端 responseTime 分位/actionDistribution/sourceDistribution/timeliness 1h 阈值/avgAlternativeCount）。100% 单测（44 测）。
- **repository**：filter 下推 DB（decisions 按 responseStatus/agentId/tenantId；judgment_records 按 decisionId/source），破 pagination.ts EAV 全量限制；listPaged limit/offset+total 下推 DB（§7.2.1#2）。
- **service**：respondDecision 状态机 + **自动生成 JudgmentRecord 审计留痕**（修复原 route respond 不留 record bug——前端按 decisionId 查 judgment-records 查不到）；getJudgmentAnalytics 全量 list+内存算（百级量可接受）。
- **route**：decisions.ts 重写薄层（serialize Date→ms），inbox 跨聚合（workorder/goal）留 route 调 cockpitRepo 标 [PLANNED]。
- **装配**：bootstrap 内联 decisionRepo+judgmentRecordRepo+decisionService（同 signal/objective 一致；bootstrap 805 行过 800 红线但未拆 bundle，逼近 1000 再抽 cockpit-bundle）。

**设计决策（用户拍板）**：① escalation 移归 E11 Orchestration（端点在 orchestration.ts + 前端 EscalationChain/EscalationPolicy 与 orchestration 同组），E10 只做 decisions+judgment_records。② judgment-analytics 对标前端 JudgmentAnalytics 全量语义（非后端旧 outcome/responseMs 简化字段），1:1 复刻避免前后端分裂。

**遗留**：① respond 自动生成 record 的 source 默认 `agent-discovery`（decision DTO 不含 source），前端 createJudgmentRecord 端点保留兼容——若前端双写导致重复 record 待核验。② getJudgmentAnalytics 全量 list（MAX_LIMIT 200），超 200 条 record 截断记 backlog。③ inbox 跨聚合聚合待 workorder/goal 实体化后下沉。

### E11（done）— 编排子系统 domain 建模

对标后端 `orchestration.ts` 现有 EAV route 字段（实现态契约），非前端 EscalationChain/CollaborationChain 设计态超前实体——前端 `cockpitApiAdapter` 全文无 orchestration DTO，前端设计态实体字段（stages/attempts/nodes/edges）与后端 route 字段（steps/currentStep/status）不对等，1:1 复刻需有损映射（守 Phase B/E10 教训 §6.3）。

- **schema**：orchestration_chains（完全结构化 name/steps jsonb 值对象数组/currentStep/status/agentId/tenantId + 4 index）+ escalations（半结构化 status/severity/triggerReason/relatedTaskId/metadata jsonb 透传/tenantId + 3 index）+ orchestration_agents（半结构化 agentId/role/status/metadata jsonb/registeredAt/tenantId + 3 index）。
- **migration**：CREATE/INDEX/INSERT 全 IF NOT EXISTS/ON CONFLICT 幂等；EAV 三 entityType 迁移，metadata = data 减已知字段剩余透传。实测无源数据迁 0 行无错，DB 验证三表 + 13 索引建对。
- **domain**：OrchestrationChain（immutable DDD，advance 状态机 currentStep++→末步 completed + status 枚举校验 + currentStep clamp）+ Escalation（transition 状态机 open→acknowledged→resolved→closed 单向禁回退 + asStatus 运行时校验拒脏值）+ OrchestrationAgent（最薄无状态机——route 无 PATCH 仅注册记录）。三实体 rehydrate 容错脏枚举。100% 单测 38 测。
- **repository**：filter 下推 DB（chains 按 status/agentId/tenantId；escalations 按 status/tenantId；agents 按 agentId/status/tenantId）破 pagination.ts EAV 全量限制；listPaged limit/offset+total 下推（§7.2.1#2）。chain 用 fromProps 严格，escalation/agent 用 rehydrate 容错（半结构化 EAV 旧数据可能脏，同 judgment-record 范式）。

- **route**：orchestration.ts 重写薄层（serialize Date→ms + parsePagedQuery，守 §12信号6）。PATCH escalation 收敛 status 必传（缺 400）+ metadata 可选。advance 保留诚实注释 [PLANNED] 真调度接 /agent/dispatch。

**设计决策**：① 对标源 = 后端 orchestration.ts 现有 route 字段（实现态契约），非前端设计态实体（cockpitApiAdapter 全文无 orchestration DTO，守 Phase B/E10 教训 §6.3）。② 诚实分层：chain 完全结构化（route 有固定字段），escalation/agent 半结构化（route body 透传无固定契约，提 status/registeredAt 强类型列 + metadata jsonb 透传剩余，不强加超前列）。③ advance 诚实假推进（手动 currentStep++），真调度接 /agent/dispatch 留 [PLANNED]。④ 三端点前端不消费（孤儿：cockpitApiAdapter 无 orchestration DTO，fetchCollaborationChainsFromAPI 调 /api/cockpit/sessions 错配，escalation 事件流由 tasks.ts 发布），实体化破贫血 + 留真调度接口。

**遗留**：① advance 真调度接 /agent/dispatch 留 [PLANNED]。② escalation/agent metadata jsonb 透传（route 无固定契约），未来契约确立后可提强类型列。③ orchestration 三端点孤儿，未来前端接通或真调度消费时再激活。

## 对后续版本的约束

2. 五子系统实体化后，EAV cockpitEntities 的兼容/迁移策略确立（灰度或一次性迁移）。
3. 涌现信号数据回流确立后，"感知神经系统"接通，上层判断/编排/评估可基于真信号而非手动录入。
4.
