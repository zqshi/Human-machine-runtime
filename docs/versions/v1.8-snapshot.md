# v1.8 Snapshot — 声明/运行 reconcile 解耦(云原生收官)

> **状态**:已归档(done,2026-06-24)
> **主题**:声明态(spec/desired)与运行态(actual)分离,reconcile controller 按 desired→actual diff 增量调和(reconcile loop)
> **前置**:v1.3-v1.7(见 v1.3-v1.7-snapshot.md)
> **架构总纲**:`docs/architecture/cloud-native-platform-design.md` §4.v1.5(原编号)

## 版本目标

instance 由单态模型(state 既是声明又是运行)升级为声明/运行分离:state=actual only,desiredState 独立;reconcile controller 按 desired→actual + specGeneration diff 增量调和(非整机重建),health-monitor 退为兜底。云原生 5 版收官。

## 交付内容摘要

| ID | 交付 |
|----|------|
| T1 | `desired_state` + `spec_generation` 字段(schema/domain/migration `migrations/instance.ts`/repository) |
| T2 | `IInstanceProvisioner.reconcile` + `getRemoteStatus` 进接口;三 provisioner 实现;补齐此前缺失的 provisioner 测试 |
| T3 | `InstanceService.reconcile` 编排 + domain 纯决策 `decideReconcileAction`;start/stop/rebuild 维护 desiredState;updateResources/policy 等 bump specGeneration(drift 信号) |
| T4 | `instance-reconciler` controller(*/5)+ bootstrap-tasks seed + bootstrap 注册 |
| T5 | health-monitor 升级:unhealthy 先 `force-reconcile` 轻量恢复,rebuild 仅兜底(30min cooldown) |
| T6 | 全量门禁 + migration 真实验证 |

## 关键技术决策

1. **reconcile 决策纯逻辑下沉 domain**:`decideReconcileAction(desired/actual/specGeneration) → noop/start/stop/reconcile`,application 层只编排(避免贫血模型 §12.1)。
2. **specGeneration 覆盖全 spec**:区别于 `version`(乐观锁)与 `agentGeneration`(仅 agent 定义维度),声明态变更(resources/policy/agentDefinition)统一 bump,驱动 reconcile drift 检测。
3. **reconcile 优先,rebuild 兜底**:reconcileSpec 失败累计 `reconcileFailures` 达阈值(3)→ rebuild;health-monitor unhealthy 先 force-reconcile 再累积 rebuild。两条兜底路径条件不同,cooldown 去重。
4. **诚实约束**:ContainerOrchestrator 上游无 resize API,reconcile 仅状态级(start);LocalProvisioner 真增量(模拟)。

## 验证标准达成

1. desired=stopped → teardown(actual 跟随)✓
2. 改 ResourceConfig → reconcile 增量(Local 真增量;Container 受上游限制)✓
3. specGeneration 变更 → diff 检测 → 调和 ✓
4. reconcile 失败 N 次 → rebuild 兜底 ✓

## 对后续版本的约束

1. **desired/actual 严格分离**:state=actual only,desiredState 独立;状态变更经 reconcile 调和。
2. **reconcile 优先于 rebuild**:轻量 spec-diff 调和优先,rebuild 仅兜底。

## 遗留问题

- **ContainerOrchestrator 扩容限制**:上游无 resize API,reconcile 仅状态级;真扩容需 rebuild 或上游支持 → 可入 backlog。
- **reconcile/rebuild 双兜底路径**:T3 编排兜底 + T5 health-monitor 兜底,条件不同不冲突;理论可能短时双 rebuild,由 cooldown 去重 + rebuild 幂等保护;可观察后视需要统一入口。
- **migration 回填修正**:首版默认回填 `desired_state='requested'` 与旧实例运行态不符(致 reconcile 失效),已补幂等 `UPDATE ... SET desired_state = state` 对齐,drift=0。
