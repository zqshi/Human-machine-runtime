## 版本目标

当前 instance 是单态模型(state 既是声明又是运行),health-monitor 是 liveness probe + 整机 rebuild(无 spec-diff)。改 resources/policy 后 actual 不自动跟随 desired——v1.5(现 v1.8)解决这个:声明/运行分离 + reconcile controller 按 desired→actual diff 增量调和(非整机重建)。

## 核查结论(Phase 1 完成)

**现状**:
- instance 单态(state),无 desiredState;version 是乐观锁非 spec 世代;agentGeneration 仅覆盖 agent 定义维度(不够)
- health-monitor:liveness probe + 连续 3 次不健康→整机 rebuild(teardown+recreate),无原地 reconcile;*/5 cron + bootstrap-tasks seed
- IInstanceProvisioner 极简(provision/teardown),无 reconcile;getRemoteStatus 外挂(仅 container)
- instance-service:start/stop/rebuild 有,无 reconcile;updateResources 改 desired 但不触发调和(核心问题)

**可复用**:scheduler 注册两段式(handler+task seed)、instance_health_snapshots 防抖、死信告警、start/stop/rebuild(重度调和 fallback)

**需新建**:desiredState + generation 字段、IInstanceProvisioner.reconcile + getRemoteStatus 进接口、InstanceService.reconcile、instance-reconciler.ts controller

## 版本验证标准

1. 改 desiredState=stopped → reconciler 调 teardown(actual 跟随)
2. 改 ResourceConfig.cpu → reconciler 增量调和(扩容,非整机重建)
3. spec generation 变更 → reconciler 检测 diff 应用
4. reconcile 失败 N 次 → fallback health-monitor rebuild(兜底)

## 对后续版本的约束

1. **desired/actual 严格分离**:state 改为 actual only,desiredState 独立;任何状态变更经 reconcile 调和
2. **reconcile 优先于 rebuild**:轻量 spec-diff 调和优先,rebuild 仅兜底(reconcile 失败/整机不健康)
