# v1.2.2 Next — 商业模式闭环

> **主题**: 把 `tenant-management`（套餐）+ `quota-management`（配额）+ `analytics`（用量统计）+ `billing`（计费骨架）打通成可运营的计费链路，支撑 Free/Pro/Team 三级定价
> **前置**: v1.2.1 完成（外部契约收尾，确保 token 用量统计可信）
> **核心价值**: HMR 从"可对外承诺 SLA"升级到"可计费运营"
> **状态**: 规划中（待 v1.2.1 完成后激活为 current）
> **版本拆分由来**: 原 v1.2.0 范围过大，按 §11 拆为 v1.2.1（外部契约）+ v1.2.2（本版本，商业模式）

---

## 版本目标

1. **计费写侧链路补全**: 当前 `billing` context 是骨架（仅 recordEvent 累加 amount），缺 quota 扣减、对账任务、invoice 生成
2. **用量报表 API + 前端**: 管理后台能看到按租户/套餐维度的用量报表（API 调用次数、实例运行时长、Token 消耗）
3. **可观测性补强**: 商业模式上线必须有用量审计 + 异常告警，否则计费不可信

## 版本验证标准

1. 租户在 Pro 套餐下创建数字员工实例，超配额时返回 `QUOTA_EXCEEDED`（已有，端到端验证）
2. 管理后台 `/admin/billing` 能看到按租户/套餐维度的用量报表
3. billing 写侧链路完整：预占 → 实扣 → 对账（日终核对 quota 余量 vs analytics 累计 vs billing 累加）
4. 用量异常（超阈值）触发通知通道告警

## 版本范围

**做什么:**
- `quota-management` 与 `analytics` 数据流贯通（实时扣减 + 日终对账）
- billing 写侧补全：consume/deduct/reserve API、对账任务
- 用量报表 API（按租户/套餐/时段聚合）
- 前端用量报表展示
- 用量异常告警（阈值 + 通知通道）

**不做什么:**
- 真实支付网关对接 → v1.3+（本版本只做用量统计 + invoice 生成，不做扣款）
- 账单 PDF 导出 → v1.3+
- 多币种支持 → v1.3+（当前固定 USD）

## 各任务实施要点

### T4 — quota ↔ analytics 贯通 `[核心风险点]`

**当前状态**（v1.2.1 起始时审计结果）:
- ✅ quota-service 通过 `tokenLookup.getUsageSummary` **读取** analytics 累计用量做配额检查
- ✅ `bootstrap.ts` 在 claudeAdapter.onTaskComplete 回调里调 `tokenUsageService.recordUsage` 与 `billingService.recordEvent`
- ❌ **没有 quota 扣减（consume/deduct/reserve）的写路径** —— 当前是"读时检查"模式
- ❌ **没有日终对账任务**

**待确认点**:
- quota 扣减改同步还是异步？
- 是否需要预占机制（task 开始时预占 → 完成时实扣 / 失败时释放）？

### T10 — billing 写侧补全

**当前状态**: `billing-service.ts` 头注释明确写"不实现（本期骨架）：账单生成、充值/退款、对账任务"。

**需要实现**:
- `consume(tenantId, type, amount, metadata)` — 消费扣减（与 recordEvent 区别：consume 触发 quota 联动）
- `reserve(tenantId, amount)` / `release(reservationId)` — 预占/释放（用于 task 开始时锁定预算）
- `reconcile(tenantId, period)` — 对账（与 analytics 累计核对差异）

## 对后续版本的约束

1. **商业模式数据流不可绕开 analytics**: 计费相关事件必须落 analytics，不可只写 quota。
2. **billing 写侧必须有对账**: 不允许只累加 event 不核对，否则长期漂移不可发现。
